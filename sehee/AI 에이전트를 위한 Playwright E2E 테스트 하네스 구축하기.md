# E2E 테스트 작성 및 관리 방법

- E2E 테스트는 UI 텍스트 하나 변경되었다고 깨지고, 원인 모르게 간헐적으로 깨지는(Flaky) 경우도 많음
- 모든 걸 테스트하는 건 비현실적 → 테스트의 대상이 되는 **가장 중요한 사용자 워크플로우**(Critical User Flows) 부터 찾아야 함
    - ex) 네이버페이 대출 서비스의 경우 대출 홈 → 약관 동의 → 휴대폰 인증 → 본인 확인 → 결과 조회 플로우, 상품 목록, 상품 상세
- 테스트 작성은 Codegen으로 : Playwright에서 제공하는 공식 도구. 페이지를 조작하는 것으로 러프한 테스트 코드 생성 가능

    끝내고 나면 우측의 코드와 무엇이 오고갔는지 떠 놓은 har 파일이 남음
    
- 생성된 코드를 에이전트를 통해 정리 : 기존 e2e/ 코드들, e2e-testing.md 파일 등을 통해 컨벤션을 학습하고 step 분리, 인증 등 초기화 로직 추가, 실제 테스트가 동작하는지 검증

# AI가 E2E 테스트 코드를 짤 순 없을까?

- 서비스 코드와 마찬가지로 E2E 테스트 코드도 AI가 직접 짜게 하는 것이 대세
- Playwright는 공식 에이전트 3개를 제공함 : Planner, Generator, Healer
    
    ```bash
    # loop 파라미터로 사용하는 AI 툴을 명시
    npx playwright init-agents --loop=claude
    ```
    
    - Planner : 코드를 탐색하고 테스트 계획서를 작성하는 에이전트
    - Generator : 계획서를 기반으로 테스트 코드를 작성하는 에이전트
    - Healer : 실패한 테스트를 진단 및 수정하는 에이전트

## Planner

코드베이스를 탐색하고, 사용자 플로우를 분석하여 테스트 계획서를 마크다운으로 작성

화면을 살펴보고 테스트 시나리오를 정리해주는 점이 핵심

## Generator

플래너가 만든 계획서를 기반으로 실행 가능한 테스트 코드로 옮기는 에이전트

정상 동작하는지 라이브로 검증

계획과 구현을 분리한 이유 : LLM 에이전트 하나에게 한꺼번에 시키면 완성도가 떨어질 수 있으므로 나눈 것.

## Healer

실패한 테스트를 자동으로 고쳐주는 에이전트

실패 지점을 다시 시도해보고, 원인을 분석해서 통과할 때까지 수정 및 재실행

각 에이전트들은 독립적으로도 순차적으로도 쓸 수도, 루프 안에서 돌릴 수도 있음 

# 세 에이전트만 설치하면 알아서 잘 될까?

아쉽게도 제대로 동작하지 않았음.

에이전트는 코드베이스의 테스트 전략, 도메인 지식, 프로젝트 구조에 대한 맥락이 없어 매번 헤맸음

- 인증 setup 절차를 건너뛰고 바로 시작
- 익숙한 API 에러(detailCode “90”의 해결 방법을 찾지 못해 한참 헤맴)
- 테스트 계정의 사용법을 몰라 매번 사람이 알려줘야 함

이를 해결하기 위해 프로젝트 맥락과 환경을 적어 줬음

- 인증 : getAuthState(시나리오) 사용, 계정 ID 하드코딩 금지
- 알려진 실패 패턴 : detailCode “90” → 인증 만료, setup 재실행
- 환경 : baseURL은 환경변수 PHASE에 따라 결정

# 에이전트가 잘 동작하기 위해 팀에서 준비한 환경

1. AGENTS.md : 코드베이스 진입 안내서. 서비스가 무엇이고 어떤 흐름으로 이뤄지는지 설명하는 코드베이스에 대한 전체적인 안내 문서
    - 핵심 플로우와 상품별 분기
    - 인증/접근 제어 구조
    - 도메인 용어 사전
2. e2e-testing.md : 테스트 작성 및 실행 가이드. 실행, 환경, 계정, 인증 캐시까지 흩어져 있는 룰을 모으는 SSOT
    - 환경별 baseURL
    - 시나리오별 테스트 계정
    - 인증 캐시와 만료 처리
3. 테스트 유틸&헬퍼 : 인증/초기화와 같이 반복되는 공통 동작을 함수로 분리
    - 인증(getAuthState)
    - 초기화(resetUserStatus)
    - prefill(prefillToPage)
4. MCP 서버 : Github, Figma, Atlassian 등 에이전트가 외부 데이터에 직접 접근할 수 있는 환경

⇒ 환경을 잘 깔아두니, AI가 Planner → Generator 흐름을 사용해 코드 작성까지 하게 됨

# 테스트 인프라 구축에 참고할 만한 결정들

## Prefill API : 테스트 간 중복 제거

- 모든 테스트 시나리오에서 약관 동의 → 휴대폰 인증까지는 공통, 이후 사용자 상태에 따라 분기
- 문제점 : 테스트마다 동일한 사전 단계를 반복하여 한 곳이 바뀌면 모든 테스트가 동시에 실패
- Playwright 공식문서에서는 각 테스트가 서로 독립적으로 실행되어야 함을 권장 → 재현하기 쉽고, 한 테스트의 실패가 다른 테스트로 줄줄이 번지지 않음
    
    ⇒ 각 테스트가 검증할 갈래의 시작점에서 출발하도록 테스트용 prefill API를 호출해서 상태 세팅(UI 단계 스킵)
    

## 외부 의존 차단 Mocking

- 네이버 인증서, 홈택스 전자증명서, 네이버 모바일 앱 연동 등 외부 의존을 그대로 호출할 때의 문제점
    - 테스트 실행 속도가 느려짐
    - 외부의 작은 변동(UI 변경, 응답 지연 등)에 우리 테스트가 영향을 받음
- Playwright 공식문서에서는 서드파티 의존성을 피하고 통제할 수 있는 것만 테스트하라고 권장
    
    ⇒ 호출 위치에 따라 외부 의존성 모킹으로 해결.
    
    - 브라우저 : page.route(네트워크 응답 가로채기)
    - 서버(SSR / BFF) : E2E용 환경변수(서버 코드에서 고정 응답 분기)

## 간헐적 실패를 예방하는 방법

- E2E 테스트는 서버/미들웨어/UI/API까지 한번에 검증, 그만큼 깨지기도 쉬움
- flaky한 테스트는 테스트에 대한 신뢰를 흔들고 제대로 된 검증 센서 역할을 못하게 됨
    
    ⇒ 머지되기 전 테스트 작성 단계에서 flaky를 피할 수 있게 고려
    
    - 하드 대기 금지 : waitForTimeout 대신 expect auto-wait(playwright의 조건 기반 대기, 필요한 만큼만 기다림)
    - 시맨틱한 셀렉터 : CSS 클래스나 인덱스 대신 role/name(디자인이 바뀌어도 덜 깨지도록)
    - burn-in 10회 : `pnpm playwright test --repeat-each=10` , 1번이라도 깨지면 PR 차단

# 실패하는 테스트를 에이전트가 고치는 방법

- Playwright 테스트 실행 시 액션, 네트워크, DOM, 콘솔 로그 등이 기록된 trace.zip 생성
- Healer 에이전트는 Playwright MCP/CLI 및 playwright-trace 스킬을 사용해서 trace.zip의 정보들을 CLI로 탐색 가능
- 진단이 끝나면 테스트를 수정하고 테스트가 통과할 때까지 작업

# 검증 플로우

1. 에이전트가 작업물 푸시
2. PR 전 Github Actions로 CI 및 e2e 실행
3. e2e 실패 시 PR 코멘트로 실패 결과를 제공하고 trace.zip을 아티팩트로 업로드
4. CI 결과 URL을 주면 테스트 결과 아티팩트를 다운받아 디버깅, 수정까지 하는 /e2e-debug 커스텀 스킬 작성, Healer 에이전트로 스킬을 실행하면 테스트를 수정하고 다시 PR에 반영
