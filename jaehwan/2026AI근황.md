## 2. 숫자 뒤의 구조적 전환

단일 모델보다 중요한 것은 **구조적 변화 3가지**다.

### ① 가격 붕괴 — 비즈니스 모델의 구조적 해체

| 모델 | 100만 입력 토큰 가격 | 비고 |
|------|---------------------|------|
| GPT-5.4 | ~$10 | 프론티어 |
| DeepSeek V3.2 | **$0.28** | GPT-5.4 성능의 ~90%, 비용 1/50 |
| Gemini 3.1 Flash Lite | **$0.25** | 100만 토큰 컨텍스트 |
| Gemini 3.1 Pro | ~$3.50 | 벤치마크 13/16 1위 |

2024년에 $100이던 것이 $20 미만으로. 이것은 점진적 하락이 아니라 **OpenAI 매출 모델을 구축한 경제학의 구조적 붕괴**다. 6개월 전 멀티에이전트 설정에 $400/월이었던 것이 2027년 초에는 $100~150이 될 전망.

### ② 에이전트가 기본값

- GPT-5.4: **내장 컴퓨터 사용(computer use)** 탑재
- Grok 4.20: **4개 에이전트 병렬 실행**
- Kimi K2.5: **100개 서브에이전트** 동시 조율
- Claude, ChatGPT, Gemini 모두 **브라우저 제어 에이전트 모드** 보유
- NIST: 2월 17일 **AI 에이전트 표준화 이니셔티브** 발족 (3월 9일 퍼블릭 인풋 마감)
- Gartner: 2026년 말까지 기업 앱의 **40%**가 태스크별 AI 에이전트 포함 (현재 5% 미만)
- RAND 연구: "AI 에이전트" 라벨의 80~90%는 아직 챗봇 래퍼

질문은 "AI가 추론할 수 있나?"가 아니라 **"AI가 책임 없이 자율 행동할 수 있나?"**

### 핵심: 모든 도구가 "에이전트" 카테고리로 수렴

2026년의 공통 아키텍처: **자동완성(빠른 타이핑) + 에이전트 모드(복잡한 작업) + MCP(확장성)**

차별화 축의 이동:
- ~~"AI가 코드를 쓸 수 있나?"~~ → **"워크플로우에 어떻게 통합되는가?"**
- 터미널 vs IDE vs 데스크톱 앱
- 얼마나 자율적인가 (백그라운드 에이전트, 스케줄링)
- 얼마나 확장 가능한가 (MCP, 커스텀 도구)
- 비용 예측 가능성 (크레딧 vs 쿼터 vs 정액)

## 4. Claude Code의 부상

Claude Code는 2025년 5월 출시 후 **8개월 만에 가장 많이 사용되는 AI 코딩 도구**가 되었다.

- DEV.to 개발자 AI 설문(2026.02): Claude Code **41%** > Copilot 38%
- 200인 미만 기업: **75%** 사용
- "가장 사랑하는" 도구: Claude Code **46%** > Cursor 19% > Copilot 9%

핵심 차별점: **전체 코드베이스 이해**. 단일 파일 자동완성이 아니라, 프로젝트 구조·의존성·패턴을 파악하고 멀티파일 수정을 자율 실행한다.

**실전 워크플로우 — 가장 생산적인 개발자들의 패턴:**

> 일상 편집(80%): **Cursor** — Supermaven 자동완성, Composer로 멀티파일 변경, Agent 모드로 기능 구현
> 복잡한 작업(20%): **Claude Code** — 대규모 리팩토링, 아키텍처 변경, 보안 감사, 크로스파일 디버깅

경험 있는 개발자는 평균 **2.3개 도구를 병행**. $40/월(Cursor Pro + Claude Code)은 절약되는 시간 대비 충분한 ROI.

**실제 사례:**
- OpenAI 내부: Codex로 **Sora Android 앱을 4명의 엔지니어가 18일에 출시** — "이 규모에서 이 속도는 처음"
- Rakuten: Claude Code로 **1,250만 줄 코드베이스**에서 vLLM 구현을 **7시간**에 완료, **99.9%** 수치 정확도

## 5. Vibe Coding → Agentic Engineering

### Vibe Coding이란?

2025년 2월, Andrej Karpathy가 만든 용어. 자연어로 원하는 것을 설명하면 AI가 만들어주는 개발 방식. "분위기(vibe)"를 전달하면 AI가 기술적 세부사항을 처리.

### 왜 프로덕션에서 실패했나

바이브 코딩된 소프트웨어를 프로덕션에 배포 → **"AI slop"** 양산:
- 표면적으로 합리적이나 에러 핸들링 부족
- 보안 취약점 내장
- 기존 기능 파괴
- 유지보수 불가능한 아키텍처

### Agentic Engineering: 2026년의 정답

Karpathy가 2026년 초 새로운 용어 제시: **agentic engineering** — AI 에이전트가 구조화된 인간의 감독 하에 코드를 계획·작성·테스트·배포하는 시스템을 설계하는 규율.

| | Vibe Coding | Agentic Engineering |
|---|-----------|-------------------|
| 코드 소유권 | AI에 위임 | **인간이 보유** |
| 생성 코드 리뷰 | 선택적 | **필수** |
| 아키텍처 | AI 기본 패턴 | **인간이 설계** |
| 적합한 용도 | 프로토타입, 1회성 도구 | **프로덕션 소프트웨어** |
| 실제 사례 | 개인 프로젝트 | TELUS(**50만 시간** 절약, 13,000 AI 솔루션), Zapier(**89%** AI 채택), Stripe(주 **1,000+** 머지 PR) |

### 사람에게 남는 역량

- **도메인 전문성** — 좋은 소프트웨어가 어떤 모습인지 아는 것
- **시스템 사고** — 컴포넌트 간 상호작용 이해 (오히려 더 중요해짐)
- **보안 인식** — 에이전트는 본질적으로 위협 모델을 모름
- **코드 읽기** — 쓰는 것보다 읽는 양이 폭발적으로 증가
- **에이전트용 프롬프트 설계** — 챗봇 프롬프트와 다름; 운영 사양(operational specification)
- **하네스 설계** — 에이전트를 생산적으로 유지하는 제약, 린터, 피드백 루프 구축

### AGENTS.md — 에이전트를 위한 규칙 파일

OpenAI가 제안, Linux Foundation 산하 AAIF가 관리. GitHub **60,000+개 리포지토리** 채택. Claude Code, Codex, Copilot 호환.

설치할 도구도, 구독도, CLI도 없다. **`AGENTS.md` 파일을 만들고 커밋하면 끝.**

```markdown
# AGENTS.md 예시
## 코딩 컨벤션
- TypeScript strict mode 필수
- 함수형 컴포넌트만 사용
```

## 6. MCP란 무엇인가

**Model Context Protocol**은 Anthropic이 2024년 11월에 오픈소스로 공개한 개방형 프로토콜이다. AI 모델과 외부 도구·데이터 소스를 연결하는 **범용 표준**.

이전: N개 모델 × M개 도구 = **N×M개 커스텀 통합** 필요.
이후: MCP 서버 한 번 만들면 → **모든 MCP 클라이언트에서 사용 가능**.

USB가 모든 기기를 컴퓨터에 연결하듯, MCP는 모든 AI를 모든 도구에 연결한다.

### 아키텍처

```
┌─────────────┐    JSON-RPC 2.0    ┌─────────────┐    실제 작업    ┌──────────┐
│  MCP Client │ ◄────────────────► │  MCP Server │ ◄──────────────► │  도구/DB  │
│  (AI 모델)   │                    │  (미들웨어)  │                  │          │
└─────────────┘                    └─────────────┘                  └──────────┘

Client: Claude, ChatGPT, Copilot, Gemini
Server: Playwright MCP, Slack MCP, GitHub MCP, Sentry MCP, Notion MCP 등
```

LSP(Language Server Protocol)에서 영감. 와이어 포맷은 JSON-RPC 2.0.

## 7. MCP의 폭발적 성장 — 실험에서 표준으로

| 시점 | 이벤트 |
|------|--------|
| 2024.11 | Anthropic이 MCP 오픈소스 공개. Python, TypeScript SDK. 초기 반응: "또 하나의 표준" |
| 2025.03 | **OpenAI 채택**. Sam Altman: "People love MCP." 전략적 결정 — 경쟁 프로토콜 대신 합류 |
| 2025.04 | **Google DeepMind** Gemini에 MCP 지원 확인. A2A(Agent2Agent) 프로토콜도 발표 |
| 2025.11 | 대규모 업데이트: 비동기 연산(Tasks API), 서버 레지스트리, OAuth 보안, Client ID Metadata Documents |
| 2025.12 | **Linux Foundation** 산하 AAIF에 기증. 공동 설립: Anthropic, Block, OpenAI. 지원: Google, Microsoft, AWS, Cloudflare, Bloomberg |
| 2026 초 | **10,000+ 서버, 월 9,700만+ SDK 다운로드**. Datadog, Stripe, Cloudflare, Notion, Sentry, Linear 등 프로덕션 서버 |
| 2026.03 | Datadog MCP 서버 GA. SDK: Python, TypeScript, Go, Kotlin, C#, Java, PHP |

경쟁 랩들이 중립 기관에 인프라를 기여할 때, **무언가 실제적인 일이 벌어지는 것**. 네트워크 효과는 복리로 증가 — 하나의 MCP 서버가 Claude, ChatGPT, Copilot, Gemini에서 **동시에** 작동.

### MCP vs A2A

- **MCP**: "에이전트가 도구에 말하는" 프로토콜
- **A2A** (Google, 2025.04): "에이전트가 에이전트에게 말하는" 프로토콜. 50+ 기업 지원.
- 함께 사용하면 **진정한 AI 커뮤니케이션 스택**

### MCP의 다음 과제: 토큰 효율성

대부분의 MCP 클라이언트가 모든 도구 정의를 컨텍스트에 미리 로드 → 수만 토큰 소비. 해결책:

- **Code Execution with MCP** (Anthropic 제안): 에이전트가 코드를 작성해 도구를 온디맨드로 검색·호출·필터링
- **Cloudflare "Code Mode"**: **98%+ 토큰 절약**
- **Skills vs MCP**: Anthropic이 Skills(경량, 온디맨드 로드) 방향으로 전략 조정. MCP와 상호 보완적.

### 프론트엔드에서 MCP가 의미하는 것

- **테스트**: AI가 자연어로 E2E 테스트 수행. "잘못된 비밀번호로 로그인하면 에러가 뜨는지 확인해줘"
- **도구 조합**: 하나의 AI 세션에서 Playwright MCP(브라우저) + Sentry MCP(에러) + Slack MCP(알림) + GitHub MCP(이슈)
- **진입 장벽 하락**: Playwright API 몰라도 자연어로 브라우저 자동화 가능
- **MCP가 테이블 스테이크**: Cursor와 Copilot이 2026년 말까지 MCP 지원 추가 전망. 커스텀 도구 확장 능력이 너무 강력해 경쟁자들이 무시 불가.

## 8. CDP — 모든 것의 기반

### CDP란?

**Chrome DevTools Protocol** — Chrome/Chromium을 프로그래밍으로 제어하는 저수준 WebSocket 프로토콜. 브라우저 DevTools를 열 때 사용하는 것이 바로 이 프로토콜. **약 300개 이상의 커맨드**가 Page, Network, DOM, Runtime 등 도메인으로 구성.

### 역사

```
2013-14  Blink이 WebKit에서 포크 → CDP 확립
2017     Headless Chrome → Puppeteer 등장 (Google, CDP 위의 Node.js 라이브러리)
2020     Playwright 등장 (Puppeteer 핵심 엔지니어들이 Microsoft로 이직)
2025~    AI 에이전트 회사들이 Playwright를 버리고 raw CDP로 회귀
```

핵심: **Puppeteer, Playwright, Selenium 4 — 모두 CDP 위에 구축된 추상화 레이어**. CDP는 기반이지 천장이 아니다.

### 추상화 vs 직접 제어

| | CDP (직접) | Playwright/Puppeteer (추상화) |
|---|-----------|------------------------------|
| **제어 수준** | 브라우저 내부 완전 접근 | 테스트용 고수준 API |
| **성능** | 네이티브 커맨드 → 빠름 | JS 인젝션 → Chromium에서 **15~20% 느림** |
| **크로스 브라우저** | Chrome 전용 | Chrome, Firefox, WebKit |
| **WebSocket 트래픽** | ~11KB | ~326KB (동일 작업) |
| **난이도** | 매우 높음 (상태 관리 직접) | 낮음 (자동 대기, 에러 핸들링) |

AI 브라우저 회사(Browser Use 등)가 Playwright를 버리고 raw CDP로 전환한 이유: 추상화가 AI 에이전트에 필요한 브라우저 내부 정보를 가림. 한 회사는 **요소 추출 속도 5배 향상**. 다만 탭 크래시, 레이스 컨디션, 상태 관리를 직접 구현해야 함 — Playwright가 해결하려던 바로 그 문제들.

### 프론트엔드 개발자를 위한 CDP 활용

```typescript
// Playwright에서 CDP 세션 열기
const cdpSession = await page.context().newCDPSession(page);

// 성능 프로파일링 도메인 활성화
await cdpSession.send("Performance.enable");
await cdpSession.send("HeapProfiler.enable");
await cdpSession.send("Profiler.enable");

// 힙 스냅샷 캡처 → 자동 메모리 누수 탐지
const snapshot = await cdpSession.send("HeapProfiler.takeHeapSnapshot");

// 네트워크 차단 (이미지 로드 없이 성능 테스트)
await cdpSession.send("Network.enable");
await cdpSession.send("Network.setBlockedURLs", {
  urls: ["*.png", "*.jpg", "*.jpeg", "*.gif"]
});

// 콘솔 로그 수집
await cdpSession.send("Log.enable");
```

활용: **자동화된 성능 테스트, 네트워크 인터셉션, 메모리 누수 탐지, 콘솔 에러 모니터링**을 CI/CD에 통합.

## 9. Playwright MCP & CLI

### Playwright MCP Server

Microsoft 공식 MCP 서버. Playwright의 브라우저 자동화를 MCP로 노출. **접근성 트리(Accessibility Tree)**로 페이지를 구조적으로 이해 — 스크린샷/비전 모델 불필요, 빠르고 결정적(deterministic).

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

VS Code, Cursor, Claude Desktop, Windsurf, Cline, Goose, Kiro, Copilot CLI에서 바로 사용. GitHub Copilot Coding Agent에서는 **자동 설정**.

### 작동 흐름

```
1. AI가 명령 전송        → "example.com을 열어줘"
2. MCP 서버가 실행       → Playwright로 실제 브라우저에서 페이지 로드
3. 구조화된 결과 반환     → URL, 로드 상태, 접근성 트리 스냅샷
4. AI가 분석 후 다음 결정 → "로그인 버튼을 클릭해"
5. 루프 반복             → 작업 완료까지
```

각 단계가 **실제 브라우저 상태**에 기반 — 예측이 아니다.

### Playwright CLI — 2026년의 게임 체인저

MCP의 문제: **토큰 소비가 크다**. 도구 스키마 + 접근성 트리를 AI 컨텍스트에 전부 로드해야 하므로.

2026년 초, Microsoft가 `@playwright/cli` 출시:

> **동일 작업에서 MCP ~114,000 토큰 vs CLI ~27,000 토큰. 약 4배 차이.**

CLI는 접근성 스냅샷과 스크린샷을 **디스크 파일로 저장**, AI의 컨텍스트에는 로드하지 않음. Claude Code, Copilot, Cursor 같은 코딩 에이전트는 파일시스템 접근이 가능하므로 이 방식이 유리.

### CLI 사용 예시

```bash
# 페이지 열기
playwright-cli open https://demo.playwright.dev/todomvc --headed

# 텍스트 입력 & 체크
playwright-cli type "Buy groceries"
playwright-cli press Enter
playwright-cli check e21    # 접근성 트리 요소 ID

# 스크린샷
playwright-cli press screenshot

# 세션 관리
playwright-cli -s=myapp open example.com
playwright-cli -s=myapp click e5
playwright-cli -s=myapp close
playwright-cli list          # 모든 세션 확인
playwright-cli close-all     # 전체 종료
playwright-cli kill-all      # 강제 종료

# 비디오 녹화
playwright-cli video-start
playwright-cli open playwright.dev
playwright-cli video-stop

# 브라우저 변경
playwright-cli install --browser=firefox
playwright-cli config --isolated --browser=firefox --headed

# 프로필 관리
playwright-cli open example.com                 # 에피머럴 (인메모리, 기본값)
playwright-cli open example.com --persistent    # 쿠키 유지
playwright-cli open example.com --profile=./my-data  # 커스텀 프로필
```

### MCP vs CLI — 선택 기준

| | Playwright MCP | Playwright CLI |
|---|---------------|----------------|
| **토큰 소비** | ~114,000 | ~27,000 (4x 절약) |
| **작동 방식** | AI 컨텍스트에 도구 스키마 + 접근성 트리 로드 | 파일시스템에 결과 저장, AI는 파일로 읽기 |
| **적합한 환경** | 파일시스템 접근 불가 (Claude Desktop 채팅 등) | 파일시스템 가능 (Claude Code, Copilot, Cursor) |
| **장점** | 설정 간단, 범용적 | 대규모 코드베이스와 병행 가능 |
| **2026 추천** | 비개발 워크플로우, 채팅 기반 | 코딩 에이전트 기반 테스트·개발 |

Microsoft 공식: **"코딩 에이전트를 쓴다면 CLI+SKILLS를 쓰라."**

### 2026년 주요 변경 사항

- **세션 생명주기 단순화**: `session-*` → `-s=` 플래그. "실행 중" or "없음" (중지 상태 삭제)
- **브라우저 프로필 기본값**: 에피머럴(인메모리). 깨끗한 시작. `--persistent` 또는 `--profile=` 으로 쿠키 유지
- **보안 강화**: 파일시스템 접근이 워크스페이스 루트로 제한. `file://` URL 기본 차단
- **워크스페이스별 데몬**: 독립 프로세스 → 프로젝트 간 간섭 방지

## 10. Browser Agent — 자율 브라우저 AI

### 기존 자동화의 근본 문제

DOM 셀렉터가 깨지고, 레이아웃이 바뀌고, 인증 플로우가 변한다. 어제 동작하던 스크립트가 오늘 조용히 실패. Browser Agent는 **AI의 추론 능력**으로 이 취약성을 해결.

### 주요 Browser Agent 비교 (2026)

| 프레임워크 | 언어 | 접근 방식 | 핵심 특징 | GitHub Stars |
|-----------|------|----------|----------|--------------|
| **Browser Use** | Python | Agent-first (완전 자율) | 자연어 목표 → LLM 매 단계 추론·실행. WebVoyager **89.1%** | **81k+** |
| **Stagehand** | TypeScript | Hybrid (Playwright 확장) | `act()` / `extract()` / `observe()` 3개 프리미티브 | 활발한 오픈소스 |
| **Skyvern** | Python | Vision-first (컴퓨터 비전) | DOM 없이 시각적 이해. 처음 보는 사이트에서 즉시 동작 | 활발한 오픈소스 |
| **Playwright MCP** | - | 구조화된 접근성 트리 | 비전 모델 불필요, 무료, Copilot 내장 | Microsoft 공식 |

### 아키텍처 차이

#### Browser Use — Agent-first (완전 자율)

```
"Amazon에서 무선 키보드 검색하고 가격 비교해줘"
  → LLM이 페이지 관찰 → 다음 액션 결정 → 실행 → 재평가 → 반복
  → 매 단계 LLM 추론 (유연하지만 토큰 비용 높음)
  → 10단계 작업 기준 ~$0.07
```

캐시된 셀렉터에 의존하지 않음. 매번 새로 추론. 레이아웃이 바뀌어도 계속 진행. 최근 Playwright를 버리고 **raw CDP로 전환** — 요소 추출 5배 빠름, 비동기 반응 능력(팝업, 캡차 등에 즉시 대응).

#### Stagehand — Hybrid (결정적 + AI)

```typescript
// 예측 가능한 단계 → 결정적 Playwright 코드
await page.goto("https://example.com/login");
await page.fill("#email", "user@test.com");

// 예측 불가능한 단계 → AI 호출
await stagehand.act("click the submit button");
const data = await stagehand.extract("get all product names and prices");
const state = await stagehand.observe("what buttons are visible?");
```

**정밀도가 필요한 곳은 코드로, DOM이 불안정한 곳만 AI로.** v3(2026.02): **액션 캐싱** — 성공한 액션을 저장, 재실행 시 LLM 호출 없이 재사용. 반복 워크플로우에서 비용 대폭 절감. 이미 Playwright를 쓰는 팀에 **가장 현실적인 출발점**.

#### Skyvern — Vision-first (컴퓨터 비전)

DOM 대신 **컴퓨터 비전 + LLM 추론**으로 페이지를 이해. 정부 포털, 보험 폼 등 접근성 구조가 불완전한 사이트에 강점. 30필드 폼을 **~90초**에 완성 (수동 12분+). 내장 2FA/CAPTCHA 해결, 프록시 네트워크.

### 인프라 레이어

| 플랫폼 | 특징 | 가격 |
|--------|------|------|
| **Browserbase** | 관리형 Chromium 플릿. 안티봇, CAPTCHA, 세션 리플레이, 프록시. Stagehand 공식 백엔드. $40M 시리즈 B(2025.06) | $20~99/월 |
| **Steel** | 오픈소스 셀프호스팅. 6,400+ Stars | 무료 |

**Browser Agent = AI 두뇌, Browserbase/Steel = 브라우저 근육.**

### 실전 시나리오

**시나리오 1: AI 기반 E2E 테스트**
```
개발자: "회원가입 플로우를 테스트해줘"
```

# Part 5. Pretext — AI 협업의 결정적 사례

---

## 11. 한 사람이 업계를 바꾼 라이브러리

2026년 3월 27일, **Cheng Lou**(React, ReScript, Midjourney 핵심 엔지니어)가 Pretext 공개. 48시간 만에 GitHub **14,000+ Stars**, X에서 **1,900만 뷰**.

### 이것이 왜 중요한가

**15KB, 의존성 제로 TypeScript 라이브러리**. 멀티라인 텍스트 측정과 레이아웃을 **DOM을 우회하여 순수 산술(pure arithmetic)로** 수행.

웹에서 텍스트 높이를 알려면 `getBoundingClientRect()`를 호출 → **동기적 리플로우(reflow)** 발생 — 브라우저가 전체 페이지 기하 구조를 재계산하는, 웹에서 **가장 비싼 연산**. Google Chrome DevTools 팀: 단일 강제 리플로우가 모바일에서 10~100ms 메인 스레드 차단.

Pretext의 해법: 브라우저 폰트 엔진을 ground truth로 사용하되, 레이아웃 계산은 **순수 수학**으로. 결과: **300~600배 성능 향상**.

### 작동 원리: 2단계 아키텍처

```
Phase 1: prepare()
  → 텍스트 토큰화 + Canvas 측정 + 캐싱 (1회 실행)
  → 500개 텍스트 블록 전처리: ~19ms

Phase 2: layout()
  → 순수 산술 레이아웃 계산 (매 프레임 실행 가능)
  → 레이아웃 계산: ~0.09ms
  → 컨테이너 너비 변경 시 layout()만 재호출 — prepare() 불필요
```

이 분리로 **120fps 텍스트 리플로우** 가능.

### 지원 범위

- **다국어**: 한국어, 중국어, 일본어, 아랍어(RTL), 태국어
- **이모지**: 브라우저 레벨 Canvas 불일치 자동 보정
- **렌더링**: DOM, Canvas, SVG, 서버사이드(예정)
- **브라우저**: Chrome, Safari, Firefox 크로스브라우저

### AI가 구체적으로 한 일

Cheng Lou는 **Claude와 Codex**로 AI 기반 검증 루프를 구축:

```
1. 레이아웃 알고리즘 작성
2. 알고리즘 출력 vs 실제 브라우저 렌더링 비교 (수천 케이스)
3. 불일치 시 AI가 알고리즘 수정
4. 반복
```

Pretext 소스의 각 브라우저 테스트 스위트: **7,000+ 케이스, 총 70,000줄+**. 1인이 수동으로 할 수 있는 규모가 아니다.

Cheng Lou: **"지옥의 깊은 곳을 기어 나왔다(crawled through depths of hell)."** AI가 있었기에 그 지옥에서 살아 나올 수 있었다.

### 이것이 증명하는 것

> **AI가 엔지니어를 대체한 것이 아니라, 가장 어렵고 즐겁지 않은 구현 병목을 돌파하도록 도운 것.**

| 사람이 한 것 | AI가 한 것 |
|-------------|-----------|
| 핵심 아이디어: "DOM 우회 순수 산술 레이아웃" | 7,000+ 테스트 반복 검증·수정 |
| 아키텍처: prepare/layout 2단계 분리 | 크로스브라우저 렌더링 불일치 패턴 탐지 |
| 다국어·RTL·이모지 규칙 정의 | 엣지 케이스 알고리즘 반복 개선 |
| "이 라이브러리가 왜 필요한지" 비전 | 구현의 지루하고 반복적인 부분 가속 |

> **강한 아이디어 + AI의 실행력 = 한 사람이 업계를 바꾸는 라이브러리를 출시**

이것이 agentic engineering의 실제 모습이다.

### Pretext 활용처

| 분야 | 설명 |
|------|------|
| **AI 생성 UI** | AI가 레이아웃 생성 시 텍스트 치수를 즉시 필요. DOM 측정은 너무 느림. Pretext가 즉시 제공 |
| **가상 스크롤링** | 수만 개 텍스트 블록 높이를 DOM 없이 사전 계산 → 부드러운 무한 스크롤 |
| **채팅 버블** | 콘텐츠에 딱 맞는 너비, 멀티라인 센터링 — CSS 트릭 없이 순수 측정 |
| **매거진 레이아웃** | 이미지·도형을 감싸는 텍스트 리플로우 60fps. NYT가 Snow Fall에 6개월 걸린 것을 몇 줄로 해결 |
| **데이터 대시보드** | 120fps 가상화 피드, 고빈도 데이터 업데이트 |

### 풀스택 경계의 소멸

- **타입 안전 풀스택**: tRPC, Prisma, Zod로 DB ~ UI 하나의 타입 시스템. 통합 버그의 전체 카테고리 제거.
- **"백엔드리스" 패턴**: Next.js Server Actions, Remix loader/action. 프론트엔드가 API 소유.
- **메타 프레임워크 표준화**: Next.js, Nuxt, SvelteKit이 라우팅·빌드·배포 추상화. "라우터를 고르는 시대" 종료.

### AI가 대체할 수 없는 것

- **아키텍처 의사결정** — "서버 컴포넌트 vs 클라이언트?" / "상태를 어디에?"
- **도메인 지식** — 결제 플로우, 의료 규정, 금융 컴플라이언스. AI는 코드를 생성하지만 비즈니스 규칙의 "왜"를 모른다.
- **사용자 경험 판단** — 정량화 불가능한 "어색함" 감지 감각
- **시스템 간 통합** — 동시성, 성능 튜닝, 분산 장애 등 비결정적 런타임
- **윤리와 접근성** — AI 생성 코드가 모든 사용자를 포함하는지 검증

### AI 과의존의 위험

- **AI 번아웃**: 과도한 위임 → 이해도 하락. 추상화와 설계의 주도권 유지 필수.
- **Hallucination**: 자신 있게 틀린 코드. 46% 경험.
- **블랙박스 의존성**: "왜 이렇게 동작하는지" 설명 못하는 코드 증가.
- **보안 사각지대**: IP 보호, 취약점 검증, 데이터 유출 리스크.

### 자동화 타임라인

| 시기 | 자동화되는 것 | 사람에게 남는 것 |
|------|-------------|----------------|
| **단기 (1~2년)** | 스캐폴딩, 컴포넌트 생성, 단위 테스트, 문서화, 버그 수정 | 아키텍처, 코드 리뷰, UX 판단 |
| **중기 (2~5년)** | 복잡한 프론트엔드 기능, API 연동, 일부 CRUD | 아키텍처, 핵심 통합, 성능 전략 |
| **장기 (5년+)** | 많은 일상적 백엔드 작업까지 | 신규 시스템 설계, 컴플라이언스, 고위험 결정 |

프론트엔드가 먼저 자동화 압력: **시각적 피드백 루프가 명확, 모듈화, 학습 데이터 풍부**. "사라진다"가 아닌 **"더 높은 수준으로 이동한다"**.

## 17. 마무리

> "프론트엔드 개발자의 미래는 코드를 얼마나 빨리 치느냐가 아니라,
> 얼마나 좋은 판단을 내리느냐에 달려 있다."

2026년 2~4월은 AI 역사상 가장 밀도 높은 90일이었다:

- **255개 이상의 모델**이 출시되고
- **7개 코딩 도구**가 에이전트 카테고리로 수렴하고
- **MCP**가 9,700만 다운로드로 업계 표준이 되고
- **Pretext** 하나가 AI 협업의 가능성을 증명하고
- **60%의 신규 코드**가 AI에 의해 생성되는 시대가 됐다

Cheng Lou는 AI와 협업해 **한 사람이 업계를 바꾸는 라이브러리**를 출시했다. Browser Agent 생태계는 **셀렉터가 깨져서 테스트가 실패하는 시대**를 끝내고 있다. Claude Code는 **8개월 만에 가장 사랑받는 코딩 도구**가 됐다.

AI가 반복 작업을 가져간 자리에, 우리는 **아키텍처, 사용자 경험, 접근성, 그리고 비즈니스 맥락**을 채워야 한다. 2026년의 프론트엔드 개발자는 코더가 아니라 **오케스트레이터**다.

도구를 두려워하지 말고, 도구 위에서 설계하자.

> 오늘 무시하는 모든 트렌드는 내일의 기술 부채가 된다.
> 오늘 일찍 습득하는 모든 스킬은, 남들이 따라잡을 때 이미 당신의 우위다.
