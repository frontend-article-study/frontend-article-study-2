# max-age:0을 박았는데 왜 옛 페이지가 보이지?

> 브라우저, 네트워크, CDN — 한 요청이 거치는 캐시들을 전부 까보기
> 

## 들어가며

점검 페이지 마이그레이션을 진행하다가 이상한 현상을 만났다.

CloudFront → Next.js로 origin을 갈아끼우려고 Route53 레코드를 변경했는데, 새 페이지가 보여야 할 자리에 옛 페이지가 그대로 떴다. HTML 응답에는 `Cache-Control: no-cache`가 박혀 있었고, `max-age: 0`까지 추가해봤지만 결과는 같았다.

응답 헤더를 까보니 `X-Cache: Hit from cloudfront`. 옛 origin이 캐시 응답을 그대로 내려주고 있었다.

처음엔 단순한 "캐시 무효화" 이슈로 봤지만, 파고들수록 그게 아니었다. **요청이 거치는 모든 층마다 캐시가 있었고, 각 층은 통제하는 도구가 달랐다.** `Cache-Control`이 손도 못 대는 캐시도 있었다.

이 글은 그 디버깅 과정에서 정리한, "한 HTTP 요청이 origin에 닿기까지 거치는 캐시들"에 대한 기록이다.

## 캐시는 한 곳이 아니다

흔히 "캐시"라고 하면 브라우저 디스크 캐시를 떠올린다. 하지만 HTTP 요청 한 번이 origin에 닿기까지는 여러 층의 캐시를 거친다.

크게 세 부류로 나뉜다.

**A. 브라우저 HTTP 캐시 (응답 본문을 들고 있음)**

1. bfcache (Back/Forward Cache)
2. Memory Cache
3. Disk Cache
4. Service Worker Cache

**B. 브라우저 네트워크 캐시 (연결 정보를 들고 있음)**

1. DNS Cache
2. Socket Pool

**C. 서버 측 CDN 캐시**

1. CloudFront 같은 엣지 캐시

여기서 핵심 포인트 하나. **`Cache-Control` 같은 HTTP 헤더가 통제할 수 있는 건 A와 C뿐이다.** B는 HTTP 레이어가 아니라 네트워크 레이어라서 HTTP 헤더가 닿지 못한다. 이번 문제의 진짜 함정이 여기에 있었다.

하나씩 살펴보자.

## A. 브라우저 HTTP 캐시 — 4종

### 1) bfcache (Back/Forward Cache)

뒤로/앞으로 가기 시 사용된다. 페이지의 메모리 스냅샷을 통째로 보관해두고, 사용자가 뒤로가기를 누르면 **HTTP 캐시조차 거치지 않고** 즉시 복원한다.

`Cache-Control` 무시. JS 실행 상태까지 그대로 살아있다. DevTools Network 탭에서 Size가 `(back/forward cache)`로 찍히면 이거다.

bfcache 비활성화는 `Cache-Control: no-store`나 `unload`/`beforeunload` 이벤트로만 가능하다. 일반적인 캐시 통제 도구는 안 통한다.

### 2) Memory Cache

탭이 살아있는 동안만 유지되는 휘발성 캐시. 같은 탭에서 같은 이미지를 여러 번 참조하면 메모리에서 바로 꺼낸다. 탭을 닫으면 사라진다.

### 3) Disk Cache

브라우저 프로필에 영구 저장. 보통 "브라우저 캐시"라고 부르는 그것. 며칠 뒤에 와도 재사용 가능하다.

**시크릿 모드에서 비활성화되는 게 바로 이 캐시다.** 시크릿은 디스크에 흔적을 남기지 않는 게 목적이라, 받은 응답을 메모리에만 보관했다가 창을 닫으면 폐기한다.

### 4) Service Worker Cache

PWA나 Workbox가 등록한 SW가 자체 Cache Storage에서 응답을 가져온다. **가장 위에서 모든 요청을 가로채기** 때문에, SW가 "옛 응답 줘"라고 말하면 다른 어떤 캐시 정책도 무력화된다.

DevTools → Application → Service Workers / Cache Storage에서 확인 가능하다.

이 4가지는 모두 **HTTP 응답 본문**을 들고 있다. `Cache-Control`, `ETag`, `Last-Modified` 같은 HTTP 헤더가 통제하는 영역이다. `Clear-Site-Data: "cache"` 헤더로도 비울 수 있다.

## B. 브라우저 네트워크 캐시 — 2종

여기서부터가 진짜 함정 구역이다.

### 5) DNS Cache

도메인 → IP 매핑을 캐싱한다. 다층 구조다.

```
브라우저 자체 DNS 캐시  ← chrome://net-internals/#dns
    ↓
OS 리졸버 캐시
    ↓
ISP DNS 서버 캐시
    ↓
권한 있는 서버 (Route 53 등)
```

각 층이 독립적으로 TTL에 따라 결과를 보관한다. Route53에 TTL 3600초로 박혀있으면, 사용자 ISP가 옛 IP를 한 시간씩 들고 있을 수 있다.

크롬은 자체 DNS 캐시를 또 따로 갖고 있다. 그래서 OS DNS를 비워도(`sudo dscacheutil -flushcache` 등) 크롬은 옛 IP를 들고 있을 수 있다. 검증·비우기는 `chrome://net-internals/#dns`에서 가능하다.

### 6) Socket Pool

이게 의외로 모르는 사람이 많다. 크롬은 한 번 맺은 TCP 연결을 닫지 않고 풀에 보관해뒀다가, 같은 origin으로 가는 다음 요청에 재사용한다. **HTTP keep-alive**의 메커니즘이다.

```
1. DNS 조회        (수십 ms)
2. TCP handshake   (1 RTT)
3. TLS handshake   (1~2 RTT, 수백 ms)
4. HTTP 요청
```

이 무거운 절차를 매 요청마다 다시 하지 않으려고 만든 최적화다. 한 페이지에 수십~수백 자원이 있으니 효과는 어마어마하다.

`chrome://net-internals/#sockets`에서 풀 상태를 볼 수 있다. **Idle sockets**가 keep-alive로 살아있는 연결들이다.

여기서 두 번째 함정. **DNS 캐시를 비워도 socket pool은 그대로다.** 풀에 살아있는 연결이 있으면 DNS 재조회를 안 한다. 이미 그 IP에 줄 연결돼 있으니 그냥 거기로 요청을 부어버린다.

```
[DNS 캐시 비웠음]   gongbiz.kr → ?
[Socket Pool]      1.2.3.4(옛 IP)와 keep-alive 연결 살아있음
   ↓
다음 요청 → DNS 재조회 없이 그냥 1.2.3.4로 감
```

이 두 캐시는 **HTTP 헤더로 통제 불가능**하다. `Cache-Control`, `Clear-Site-Data`, 그 어떤 것도 못 건드린다. 통제 도구가 따로 있다.

| 캐시 | 통제 방법 |
| --- | --- |
| DNS Cache | DNS 레코드의 TTL 설정 |
| Socket Pool | `Connection: close` 헤더, 서버 idle timeout |

## C. CDN 캐시

서버 쪽, 모든 사용자에게 공유되는 캐시. 누가 처음 요청해서 origin에서 채워두면, 다음 사람들은 모두 여기서 받는다.

브라우저 캐시와 다른 점은:

- **공유**됨 (시크릿 모드에서도 hit 가능)
- 응답 헤더로 식별 가능: `X-Cache: Hit from cloudfront`, `Age`, `Via`, `X-Amz-Cf-Pop`
- `Cache-Control`로 통제 가능 (브라우저 캐시와 같은 메커니즘)
- 즉시 비우려면 invalidation API 호출

## 전체 요청 플로우

요청 한 번이 어떤 길을 가는지 따라가보자.

1. **브라우저 HTTP 캐시 확인** — bfcache → Memory → Disk → Service Worker 순. 히트면 즉시 응답.
2. 미스면 네트워크 요청 시작.
3. **DNS Cache 확인** — `gongbiz.kr → IP`를 알고 있나? 모르면 OS → ISP → 권한 서버까지 묻는다.
4. **Socket Pool 확인** — 그 IP에 keep-alive 연결이 있나? 있으면 재사용, 없으면 TCP+TLS handshake.
5. HTTP 요청 송출.
6. **CloudFront edge 도달** — 자기 캐시에 신선한 응답이 있나? 있으면 그대로 반환 (`X-Cache: Hit`).
7. 미스면 origin으로 (S3 또는 Next.js).
8. **Origin 응답 생성.**
9. 역방향으로 응답이 돌아오며 각 층에 캐시됨.

여기서 `Cache-Control: no-cache`가 작동하는 건 1번과 6번뿐이다. 나머지는 못 건드린다. 그리고 핵심 포인트 — **`no-cache`는 캐시를 비우는 게 아니라 검증을 강제할 뿐이다.** ETag가 같으면 304가 떨어지고 브라우저는 자기 디스크 캐시의 옛 본문을 그대로 사용한다.

```
Cache-Control: no-cache    →  매번 검증 → 304면 캐시 사용
max-age: 0                 →  매번 검증 → 304면 캐시 사용 (사실상 no-cache와 동일)
no-store                   →  아예 저장 금지 (이게 진짜 "캐시 안 함")
```

## 이번에 생긴 문제

상황을 다시 정리하면:

- Route53 레코드를 옛 origin에서 새 origin으로 변경
- 응답에 `Cache-Control: no-cache` 박힘
- 그런데 사용자에겐 옛 페이지가 계속 보임

원인은 **두 층에 동시에** 있었다.

**원인 1 — DNS Cache가 옛 IP를 들고 있었다**

Route53 레코드 TTL이 길게 잡혀있어서 ISP/OS/브라우저 단에 옛 CloudFront IP가 캐싱돼 있었다. 사용자가 새 페이지를 요청해도 그 요청이 옛 origin으로 갔다.

**원인 2 — Socket Pool이 옛 연결을 재사용했다**

설령 DNS가 새로 조회됐더라도, 이미 keep-alive로 살아있는 옛 IP의 연결이 풀에 있으면 그걸 재사용한다.

**`Cache-Control: no-cache`는 이 두 층에 어떤 영향도 못 준다.** HTTP 헤더는 HTTP 캐시(브라우저 본문 + CDN)만 통제한다. 네트워크 계층은 그 위에서 작동하기 때문에 HTTP 헤더가 닿지 않는다. 이게 핵심 통찰이었다.

증거 — 응답 헤더의 `X-Cache: Hit from cloudfront`. 옛 CloudFront에서 응답이 오고 있다는 뜻. 즉 요청이 새 Next.js origin에 도달조차 못 하고 있었다.

## 해결 — 두 가지 시나리오

### 디버깅 시 (개발자 한 명)

```
1. chrome://net-internals/#dns    → Clear host cache
2. chrome://net-internals/#sockets → Flush socket pools
3. 페이지 재방문
```

순서가 중요하다. Socket을 먼저 비우면 DNS는 살아있어서 같은 IP로 또 연결된다. **DNS 먼저 비우고 socket 비워야** 다음 요청이 DNS 재조회 → 새 IP로 새 소켓을 강제로 맺는다.

이번 케이스에선 Flush socket pools로 해결됐다. 이미 옛 IP의 연결이 풀에 살아있던 게 직접 원인이었다.

### 운영 시 (실 사용자)

사용자가 일일이 DevTools를 쓸 수는 없다. 진짜 답은 따로 있다.

**메인 솔루션 — DNS TTL을 사전에 낮추기**

마이그레이션 48시간 전부터 Route53 TTL을 60초로 낮춘다. 그러면 사용자들의 DNS 캐시가 1분 단위로 갱신된다.

**핵심은 사전 준비다.** 마이그레이션 한가운데서 TTL을 바꿔도 이미 늦었다 — 옛 TTL이 사용자에게 박힌 상태라 그 시간을 기다려야 한다. 표준 운영 패턴은 이렇다:

```
T-48시간:  TTL을 3600 → 60으로 낮춤
T-0:       Route53 레코드 새 IP로 변경
T+1시간:   트래픽 새 origin으로 안정화 확인
T+24h:     TTL 3600으로 복구
```

**보조 솔루션 — Connection: close**

마이그레이션 시기에만 origin 응답에 `Connection: close` 헤더를 박는다. 이 헤더를 받은 클라이언트는 요청이 끝나면 socket을 즉시 종료한다. 다음 요청은 새 연결 → 새 DNS 조회.

평상시엔 keep-alive를 끄면 성능이 폭망이지만, 점검 페이지처럼 일회성 트래픽이면 무관하다.

## 점검 페이지 베스트 프랙티스 (보너스)

이왕 점검 페이지 얘기가 나온 김에 같이 챙길 것들.

```
HTTP/1.1 503 Service Unavailable
Retry-After: 600
Cache-Control: no-cache, must-revalidate
Connection: close
```

- **503 + Retry-After** — 200으로 내려주면 검색엔진이 점검 페이지를 정상 콘텐츠로 인덱싱한다. 503으로 "잠깐 안 됨"임을 알려야 인덱스가 보존된다.
- **no-cache + must-revalidate** — 점검이 끝난 후 옛 점검 페이지가 디스크 캐시에 남아있지 않게.
- **Connection: close** — stale connection 방지.

## 그런데 시크릿 창에선 왜 됐을까?

디버깅 중에 던졌던 좋은 질문. 정확히 말하면 **시크릿 창은 "새로 통신"하는 게 아니라, 일반 창과 분리된 자기만의 깨끗한 컨텍스트에서 시작**한다. 결과적으로 새 통신이 일어나긴 하지만, 메커니즘이 좀 미묘하다.

### 시크릿 창은 무엇이 분리되는가

크롬 시크릿(Incognito) 모드는 **격리된 프로필**처럼 동작한다. 일반 창의 다음 항목들과 완전히 분리된다.

| 항목 | 일반 창 | 시크릿 창 |
| --- | --- | --- |
| Disk Cache | 공유 저장소 | **별도, 메모리만** |
| Memory Cache | 일반 풀 | **별도** |
| DNS Cache | 일반 풀 | **별도** |
| Socket Pool | 일반 풀 | **별도** |
| Cookies | 일반 jar | **별도** |
| LocalStorage / SessionStorage | 영구 | 휘발성 |
| Service Worker | 등록됨 | **무시됨** |

**핵심: 일반 창과 시크릿 창은 캐시·소켓을 공유하지 않는다.**

시크릿 창을 새로 열면 DNS 캐시, Socket Pool 모두 **빈 상태에서 시작**한다. 이전 일반 창에서 옛 IP에 맺어둔 socket이 있어도 시크릿은 그걸 못 본다.

### 한 가지 함정 — OS DNS는 공유된다

미묘한 부분인데, **시크릿 창의 DNS 캐시는 분리돼 있지만, OS 레벨 DNS는 모든 프로세스가 공유**한다.

```
시크릿 크롬 DNS 캐시 (분리)
   ↓ 미스 시
OS resolver 캐시 (공유) ← 일반 크롬도 여기 봄
   ↓ 미스 시
ISP DNS → Route53
```

그렇다면 이번 케이스에서 시크릿이 정상 동작한 이유는?

**진짜 원인은 DNS가 아니라 Socket Pool이었을 가능성이 가장 크다.**

- 일반 창: DNS는 이미 새 IP로 갱신됐을 수도 있는데, **옛 IP에 맺어둔 socket이 풀에 살아있어서** 그걸 재사용
- Flush socket pools → 옛 socket 죽음 → 다음 요청은 새 DNS → 새 origin 도달
- 시크릿: socket pool이 처음부터 비어있음 → 새 origin 도달

**즉 socket pool 격리가 시크릿이 정상 보였던 진짜 이유**다.

### 시크릿이 디버깅 도구로 강력한 이유

시크릿 창은 사실상 **"브라우저 로컬 상태를 전부 초기화한 새 사용자"**를 시뮬레이션한다.

- 시크릿에서 정상 보이면 → 브라우저 측 캐시 문제 (Disk/SW/Socket/브라우저 DNS)
- 시크릿에서도 똑같이 망가져 있으면 → 더 깊은 층(OS DNS/CDN/Origin) 문제

이번처럼 시크릿이 정상이었다 = **브라우저 로컬 상태(특히 socket pool) 문제 확정**. 그러니 운영 사용자의 체감은 사용자별로 다를 수밖에 없다. 사용자 A의 브라우저는 옛 socket을 들고 있어서 망가졌지만, 사용자 B는 마침 socket이 idle timeout으로 죽어서 정상이었을 수도 있다. **사용자별 체감이 갈리는, 디버깅 난도가 높은 버그였다.**

운영 시 `Connection: close`나 사전 DNS TTL 조정이 중요한 이유가 이것이다. 모든 사용자의 브라우저 로컬 상태를 강제로 초기화시킬 수 있는 방법이 그것뿐이다.

## 디버깅 체크리스트

다음에 비슷한 상황을 만났을 때를 위해 정리.

1. **시크릿 모드에서 정상이면** → 브라우저 로컬 캐시(Disk/SW) 문제. 일반 모드에서 캐시 비우면 끝.
2. **시크릿에서도 옛 페이지** → 더 깊은 층(DNS/Socket/CDN). 아래 단계로.
3. **`dig <도메인>`으로 IP 확인** → DNS 전파 상태. 옛 IP면 DNS 캐시 의심.
4. **응답 헤더의 `Server`, `X-Cache`, `Via` 확인** → 어느 origin에 도달했는지.
    - `Server: AmazonS3` + `X-Cache: Hit from cloudfront` → 아직 옛 CDN 경로
    - 새 서버 시그니처 → 새 origin 도달 OK
5. **ETag/Last-Modified가 배포 후 바뀌었는지** → origin이 실제로 갱신됐는지.
6. **`chrome://net-internals/#dns`, `#sockets` 확인** → 로컬 네트워크 캐시.

## 정리

> "캐시"라는 단어는 7개의 다른 메커니즘을 한 단어로 부르고 있다.
> 

| 층 | 위치 | 통제 방법 |
| --- | --- | --- |
| bfcache | 브라우저 메모리 | `Cache-Control: no-store`, `unload` 이벤트 |
| Memory Cache | 브라우저 메모리 | `Cache-Control` |
| Disk Cache | 브라우저 디스크 | `Cache-Control` |
| Service Worker | 브라우저 SW | SW 코드 + `Clear-Site-Data` |
| DNS Cache | OS·브라우저·ISP | DNS TTL |
| Socket Pool | 브라우저 | `Connection: close`, 서버 idle timeout |
| CDN Cache | 엣지 서버 | `Cache-Control`, invalidation API |

`Cache-Control`은 만능이 아니다. **어떤 층의 캐시를 통제할지부터 정해야** 올바른 도구를 고를 수 있다.

이번 디버깅에서 얻은 가장 큰 교훈은 **"HTTP 캐시"와 "네트워크 캐시"의 구분**이었다. HTTP 헤더는 HTTP 레이어만 통제한다. 그 아래 레이어(DNS, TCP)는 별도의 도구로 관리해야 한다는 사실. 마이그레이션처럼 origin이 바뀌는 이벤트에선 이 구분을 놓치면 며칠을 헤맨다.

다음에 또 비슷한 상황을 만나면, 이 7개 층을 위에서부터 차례로 의심해보면 된다.
