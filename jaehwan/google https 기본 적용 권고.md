# HTTPS 기본 적용

---

지금으로부터 1년 후, 2026년 10월 Chrome 154 출시와 함께, 우리는 Chrome의 기본 설정을 변경하여 "항상 보안 연결 사용(Always Use Secure Connections)"을 활성화할 것입니다. 이는 Chrome이 HTTPS가 없는 공개 사이트에 처음 접속하기 전에 사용자의 허락을 요청한다는 것을 의미합니다.

> *"항상 보안 연결 사용" 설정은 HTTPS가 없는 사이트에 접속하기 전에 사용자에게 경고합니다*

Chrome 보안팀의 사명은 링크를 클릭하는 것을 안전하게 만드는 것입니다. 안전하다는 것의 일부는 사용자가 URL을 입력하거나 링크를 클릭할 때, 브라우저가 사용자가 의도한 곳으로 도착하도록 보장하는 것을 의미합니다. 링크가 HTTPS를 사용하지 않을 때, 공격자는 탐색을 탈취하여 Chrome 사용자가 임의의 공격자가 제어하는 리소스를 로드하도록 강제할 수 있으며, 사용자를 멀웨어, 표적 공격, 또는 사회공학 공격에 노출시킬 수 있습니다. 이러한 공격은 가상의 것이 아닙니다—탐색을 탈취하는 소프트웨어는 쉽게 구할 수 있으며, 공격자들은 이전에 표적 공격에서 사용자 기기를 침해하기 위해 안전하지 않은 HTTP를 사용한 적이 있습니다.

공격자는 단 하나의 안전하지 않은 탐색만 있으면 되기 때문에, 많은 사이트가 HTTPS를 채택했다는 것에 대해 걱정할 필요가 없습니다—어떤 단일 HTTP 탐색이든 발판을 제공할 수 있습니다. 더 나쁜 것은, 오늘날 많은 평문 HTTP 연결이 사용자에게 완전히 보이지 않는다는 것입니다. HTTP 사이트가 즉시 HTTPS 사이트로 리디렉션할 수 있기 때문입니다. 이는 사용자에게 위험이 발생한 후 Chrome의 "보안되지 않음" URL 바 경고를 볼 기회를 주지 않으며, 애초에 스스로를 안전하게 지킬 기회도 주지 않습니다.

이 위험을 해결하기 위해, 우리는 2022년에 "항상 보안 연결 사용" 설정을 옵트인 옵션으로 출시했습니다. 이 모드에서 Chrome은 모든 연결을 HTTPS로 시도하고, HTTPS를 사용할 수 없는 경우 사용자에게 우회 가능한 경고를 표시합니다. 우리는 또한 이전에 HTTPS를 기본값으로 전환하려는 의도를 논의한 바 있습니다. 이제 모든 사용자에게 기본적으로 "항상 보안 연결 사용"을 활성화할 때가 왔다고 생각합니다.

---

## HTTPS 채택 현황

10년 이상 동안, Google은 Chrome에서 HTTPS를 사용하는 탐색의 비율을 추적하는 HTTPS 투명성 보고서를 발행해 왔습니다. 보고서의 처음 몇 년 동안, 숫자는 인상적인 상승을 보였습니다. 2015년 약 30-45%에서 시작하여, 2020년경 약 95-99% 범위에 도달했습니다. 그 이후로 진전은 대체로 정체되었습니다.

> *메인 프레임 페이지 로드의 백분율로 표현된 HTTPS 채택률*

이러한 상승은 웹 보안의 엄청난 개선을 나타내며, HTTPS가 이제 성숙하고 널리 보급되었음을 보여줍니다. 이 수준의 채택이 남은 안전하지 않은 HTTP에 대해 더 강력한 완화 조치를 고려할 수 있게 만들어 줍니다.

---

## 균형 잡힌 접근 방식

95% HTTPS가 문제가 대부분 해결되었음을 의미하는 것처럼 처음에는 보일 수 있지만, 진실은 몇 퍼센트 포인트의 HTTP 탐색도 여전히 많은 탐색이라는 것입니다. HTTP 탐색이 대부분의 Chrome 사용자에게 정기적으로 발생하기 때문에, 모든 HTTP 탐색에 대해 경고하는 순진한 접근 방식은 상당히 방해가 될 것입니다. 동시에, 정체가 보여주듯이, 아무것도 하지 않으면 이 위험이 무기한 지속될 것입니다. 이러한 위험의 균형을 맞추기 위해, 우리는 사용자에게 경고가 유발할 수 있는 잠재적 성가심을 제한하면서 웹이 더 안전한 기본값으로 이동하도록 도울 수 있는 조치를 취했습니다.

우리가 사용자에 대한 위험의 균형을 맞추는 한 가지 방법은 Chrome이 동일한 사이트에 대해 과도하게 경고하지 않도록 하는 것입니다. "항상 보안 연결 사용" 설정의 모든 변형에서, 사용자가 정기적으로 안전하지 않은 사이트를 방문하는 한, Chrome은 해당 사이트에 대해 반복적으로 경고하지 않습니다. 이는 50회 탐색 중 1회에 대해 경고하는 대신, Chrome이 사용자가 HTTPS를 사용하지 않고 새로운(또는 최근에 방문하지 않은) 사이트를 방문할 때만 경고한다는 것을 의미합니다.

## HTTP 트래픽의 주요 원인: 사설 사이트

이 문제를 더 해결하기 위해, 어떤 종류의 트래픽이 여전히 HTTP를 사용하는지 이해하는 것이 중요합니다. 안전하지 않은 HTTP의 가장 큰 기여자이며, 플랫폼 간 변동의 가장 큰 기여자는 사설 사이트에 대한 안전하지 않은 탐색입니다. 위의 그래프는 example.com과 같은 공개 사이트와 192.168.0.1과 같은 로컬 IP 주소, 단일 레이블 호스트명, intranet/과 같은 단축 링크 등 사설 사이트에 대한 탐색을 모두 포함합니다. 공개 사이트에 대해 Chrome이 신뢰하는 HTTPS 인증서를 무료로 쉽게 얻을 수 있지만, 사설 사이트에 대한 HTTPS 인증서를 얻는 것은 불행히도 여전히 복잡합니다. 이는 사설 이름이 "비고유"하기 때문입니다—사설 이름은 다른 네트워크에서 다른 호스트를 참조할 수 있습니다. 인증 기관이 검증하고 인증서를 발급할 192.168.0.1의 단일 소유자는 없습니다.

사설 사이트에 대한 HTTP 탐색은 여전히 위험할 수 있지만, 공격자가 이러한 HTTP 탐색을 이용할 수 있는 방법이 적기 때문에 일반적으로 공개 사이트 대응물보다 덜 위험합니다. 사설 사이트의 HTTP는 집 와이파이나 기업 네트워크처럼 로컬 네트워크에 있는 공격자만이 악용할 수 있습니다.

사설 사이트에 대한 탐색을 제외하면, 분포는 플랫폼 간에 훨씬 더 촘촘해집니다. 특히, Linux는 분석을 공개 사이트로만 제한하면 84% HTTPS에서 거의 97% HTTPS로 뛰어오릅니다. Windows는 95%에서 98% HTTPS로 증가하고, Android와 Mac 모두 99% HTTPS 이상으로 증가합니다.

사설 사이트에 대한 HTTP가 나타내는 감소된 위험을 인정하여, 작년에 우리는 공개 사이트만을 위한 "항상 보안 연결 사용"의 변형을 도입했습니다. 사설 사이트에 자주 접속하는 사용자(예: 기업 환경에 있는 사람들이나 웹 개발자)의 경우, 사설 사이트에 대한 경고를 제외하면 해당 사용자가 보게 될 경고의 양이 상당히 줄어듭니다. 동시에, 사설 사이트에 자주 접속하지 않는 사용자에게 이 모드는 보호의 작은 감소만을 도입합니다. 이것이 우리가 내년에 모든 사용자에게 기본적으로 활성화하려는 변형입니다.

> *"항상 보안 연결 사용", chrome://settings/security에서 사용 가능*

## 실험 결과

Chrome 141에서, 우리는 소수의 사용자에게 기본적으로 공개 사이트에 대해 "항상 보안 연결 사용"을 활성화하는 실험을 수행했습니다. 우리는 이 설정이 사용자에게 과도한 경고로 부담을 주지 않으면서 더 안전하게 유지한다는 기대를 검증하고 싶었습니다.

실험 데이터를 분석한 결과, 어떤 사용자가 보는 경고 수가 탐색의 3%보다 상당히 낮다는 것을 확인했습니다—실제로, 중간값 사용자는 주당 1회 미만의 경고를 보고, 95번째 백분위수 사용자는 주당 3회 미만의 경고를 봅니다.

"항상 보안 연결 사용"이 기본값이 되고 추가 사이트가 HTTP에서 마이그레이션하면, 실제 경고 양은 지금보다 훨씬 더 낮아질 것으로 예상합니다. 실험과 병행하여, 우리는 가장 많은 HTTP 탐색을 담당하는 여러 회사에 연락했으며, 그들이 Chrome 154의 변경 전에 HTTP에서 마이그레이션할 수 있을 것으로 기대합니다. 이러한 조직들 중 다수에게, HTTPS로 전환하는 것은 불균형적으로 어렵지 않지만, 단순히 관심을 받지 못했습니다. 예를 들어, 이러한 사이트들 중 다수는 즉시 HTTPS 사이트로 리디렉션하는 탐색에만 HTTP를 사용합니다—이전에는 사용자에게 완전히 보이지 않았던 안전하지 않은 상호작용입니다.

## 로컬 네트워크 접근을 위한 HTTP 사용 사례

HTTP의 또 다른 현재 사용 사례는 로컬 네트워크의 기기에 접근할 때 혼합 콘텐츠 차단을 피하는 것입니다. 위에서 논의한 바와 같이, 사설 주소는 비고유 이름의 소유권을 검증하는 어려움 때문에 신뢰할 수 있는 HTTPS 인증서가 없는 경우가 많습니다. 이는 대부분의 로컬 네트워크 트래픽이 HTTP를 통해 이루어지며, HTTPS 페이지에서 시작할 수 없다는 것을 의미합니다—HTTP 트래픽은 안전하지 않은 혼합 콘텐츠로 간주되어 차단됩니다. 로컬 네트워크에 접근해야 하는 일반적인 사용 사례 중 하나는 로컬 네트워크 기기를 구성하는 것입니다. 예를 들어, 제조업체가 config.example.com에 구성 포털을 호스팅하고, 그 다음 로컬 기기에 요청을 보내 구성할 수 있습니다.

이전에는 이러한 유형의 페이지가 혼합 콘텐츠 차단을 피하기 위해 HTTPS 없이 호스팅되어야 했습니다. 그러나, 우리는 최근 로컬 네트워크 접근 권한을 도입했습니다. 이는 사이트가 동의 없이 사용자의 로컬 네트워크에 접근하는 것을 방지하지만, 권한이 부여되면 HTTPS 사이트가 로컬 네트워크에 대한 혼합 콘텐츠 검사를 우회할 수 있게도 합니다. 이는 이러한 도메인을 HTTPS로 마이그레이션하는 것을 가능하게 할 수 있습니다.

## 출시 일정

우리는 2026년 10월 Chrome 154 출시와 함께 공개 사이트 변형의 "항상 보안 연결 사용" 설정을 기본적으로 활성화할 것입니다. 모든 사용자에게 기본적으로 활성화하기 전에, 2026년 4월에 출시되는 Chrome 147에서, Chrome에서 Enhanced Safe Browsing 보호를 옵트인한 10억 명 이상의 사용자에게 공개 사이트 변형의 항상 보안 연결 사용을 활성화할 것입니다.

이 전환이 대부분의 사용자에게 비교적 고통 없이 진행될 것이라는 것이 우리의 희망이자 기대이지만, 사용자는 "항상 보안 연결 사용" 설정을 비활성화하여 경고를 끌 수 있을 것입니다.

## 웹사이트 개발자 및 IT 전문가를 위한 권고

웹사이트 개발자이거나 IT 전문가이고, 이 기능의 영향을 받을 수 있는 사용자가 있다면, 마이그레이션해야 할 사이트를 식별하는 데 도움이 되도록 오늘 "항상 보안 연결 사용" 설정을 활성화하는 것을 매우 강력히 권장합니다. IT 전문가들은 경고가 표시되는 상황, 이를 완화하는 방법, 그리고 Chrome 클라이언트를 관리하는 조직(예: 기업이나 교육 기관)이 해당 조직의 필요를 충족하는 올바른 경고를 Chrome이 표시하도록 보장할 수 있는 방법을 더 잘 이해하기 위해 우리의 추가 리소스를 읽는 것이 유용할 수 있습니다.

## 향후 계획

안전하지 않은 공개 사이트에 대한 경고가 웹 보안을 위한 중요한 진전을 나타낸다고 우리는 믿지만, 아직 해야 할 일이 더 있습니다. 미래에, 우리는 특히 로컬 네트워크 사이트에 대한 HTTPS 채택의 장벽을 더욱 줄이기 위해 노력하기를 희망합니다. 이 작업은 앞으로 훨씬 더 강력한 HTTP 보호를 가능하게 해줄 것입니다.
