# 프롬프트를 잘 쓰는 개발자에서, 루프를 설계하는 개발자로

https://addyo.substack.com/p/loop-engineering
위 아티클을 프론트엔드 개발자들에게 잘 와닿는 방식으로 재구성 했습니다.

---

AI 코딩 에이전트를 쓰는 방식이 빠르게 바뀌고 있다.

지금까지는 주로 개발자가 에이전트와 대화를 잘하는 것이 중요했다. 적절한 맥락을 전달하고, 작업을 작게 나누고, 결과를 확인한 뒤 다음 프롬프트를 입력했다.

예를 들면 이런 식이다.

> 이 컴포넌트를 리팩터링해줘.
>
> 테스트도 추가해줘.
>
> 방금 수정한 부분에서 사이드 이펙트가 없는지 다시 검토해줘.
>
> 린트 에러도 해결해줘.

개발자가 작업의 흐름을 계속 붙잡고 있는 방식이다.

그런데 최근에는 한 단계 위의 접근이 등장하고 있다. 개발자가 매번 에이전트에게 다음 행동을 지시하는 대신, **에이전트가 스스로 다음 작업을 찾고, 수행하고, 검증하도록 반복 구조를 설계하는 것**이다.

이것을 **Loop Engineering**, 즉 루프 엔지니어링이라고 부른다.

---

## 1. 프롬프트 엔지니어링 다음에 등장한 것

프롬프트 엔지니어링의 핵심은 하나의 요청을 더 잘 전달하는 것이다.

반면 루프 엔지니어링의 핵심은 요청을 반복해서 전달하는 **시스템 자체를 설계하는 것**이다.

예를 들어 기존에는 개발자가 직접 이런 순서로 작업했다.

```
CI 실패 알림 확인
      ↓
개발자가 로그를 복사
      ↓
에이전트에게 “원인을 분석해줘”라고 요청
      ↓
에이전트 답변 확인
      ↓
개발자가 “그럼 수정해줘”라고 요청
      ↓
수정 결과 확인
      ↓
개발자가 테스트 실행
      ↓
실패하면 로그를 다시 전달하고 추가 요청
```

루프 엔지니어링에서는 이 과정을 하나의 반복 구조로 만든다.

```
매일 오전 CI 실패 목록을 확인한다.

실패 원인이 명확하고 수정 범위가 작다면:
1. 별도의 작업 폴더와 브랜치를 만든다.
2. 에이전트가 코드를 수정한다.
3. 테스트와 린트를 실행한다.
4. 실패하면 로그를 읽고 다시 수정한다.
5. 최대 3회까지 반복한다.
6. 다른 에이전트가 변경 사항을 검토한다.
7. 모든 검증을 통과하면 Draft PR을 만든다.

수정 범위가 크거나 3회 안에 해결하지 못하면:
- 직접 수정하지 않고 사람에게 보고한다.
```

개발자가 모든 단계에 직접 개입하는 것이 아니라, **반복 가능한 판단과 실행을 시스템으로 외부화**하는 것이다.

여기서 중요한 점은 개발자의 역할이 사라지는 것이 아니다.

개발자가 일일이 지시하던 위치에서 벗어나, 어떤 작업을 자동화해도 되는지, 어떤 조건을 만족해야 완료로 볼 것인지, 어느 지점에서 사람이 개입해야 하는지를 결정하는 역할로 이동한다.

---

## 2. 루프를 구성하는 여섯 가지 요소

원문에서는 루프 엔지니어링을 구성하는 요소를 다섯 가지 기능과 하나의 기억 장치로 설명한다.

- Automations
- Worktrees
- Skills
- Plugins and Connectors
- Sub-agents
- Memory

각각을 프론트엔드 개발자의 업무에 대입해서 살펴보자.

---

## 3. Automations: 반복 작업을 깨우는 심장 박동

- Automations?
  Automations는 루프를 일정한 주기나 조건에 따라 다시 실행하는 장치다. Codex에서는 공식 기능명으로도 사용되지만, 더 넓게 보면 스케줄 실행과 이벤트 기반 자동화를 포괄하는 개념이다.

루프가 일반적인 에이전트 실행과 다른 이유는, 한 번 실행되고 끝나지 않기 때문이다.

Automation은 일정한 주기나 조건에 따라 반복적으로 작업을 시작한다.

예를 들면 다음과 같은 작업이 있다.

- 매일 오전 전날 발생한 CI 실패를 분석한다.
- 최근 커밋에서 테스트가 부족한 영역을 찾는다.
- 매주 번들 크기 변화량을 확인한다.
- Lighthouse 점수가 일정 기준 아래로 떨어진 페이지를 찾는다.
- 디자인 시스템을 우회해서 작성된 스타일 코드를 탐지한다.
- 에러 트래킹 도구에서 반복적으로 발생하는 오류를 분류한다.

이전에도 GitHub Actions, cron, 스크립트를 조합하면 비슷한 자동화를 만들 수 있었다.

차이는 자동화가 단순히 명령어를 실행하는 수준을 넘어, **상황을 읽고 우선순위를 판단하는 에이전트와 연결되기 시작했다는 것**이다.

예를 들어 기존 스크립트는 번들 크기가 10% 증가했다는 사실을 알려줄 수 있다.

에이전트가 포함된 루프는 한 단계 더 나아갈 수 있다.

> 번들 크기가 증가한 원인은 새로 추가된 차트 라이브러리입니다.
>
> 특정 페이지에서만 사용되므로 동적 import로 전환할 수 있습니다.
>
> 수정안을 별도 브랜치에 작성했고, 빌드 결과 번들 크기가 감소했습니다.
>
> 검토가 필요한 PR은 여기에 있습니다.

Automation은 일을 대신 끝내는 기능이라기보다, **사람이 매번 찾아다녀야 했던 문제를 먼저 수면 위로 끌어올리는 기능**에 가깝다.

## 4. Worktrees: 병렬 실행이 충돌로 끝나지 않게 만들기

에이전트를 여러 개 실행하면 금방 문제가 생긴다.

한 에이전트는 공통 컴포넌트를 수정하고, 다른 에이전트는 같은 파일에서 타입 에러를 해결하고 있을 수 있다. 두 작업이 같은 체크아웃에서 진행되면 변경 사항이 섞인다.

이 문제를 해결하는 대표적인 방법이 Git Worktree다.

Worktree는 하나의 저장소에서 별도의 작업 디렉터리를 만들어 각 작업을 독립적인 브랜치에서 수행하게 한다.

예를 들어 다음 작업을 동시에 실행할 수 있다.

- Agent A: 결제 페이지 접근성 개선
- Agent B: Storybook 빌드 오류 수정
- Agent C: 사용하지 않는 feature flag 제거
- Agent D: 테스트 커버리지가 낮은 유틸 함수 보완

각 에이전트는 독립적인 작업 공간을 사용한다. 따라서 파일이 물리적으로 충돌하지 않는다.

하지만 여기서 주의할 점이 있다.

Worktree는 기계적인 충돌을 줄여줄 뿐이다. 병렬 실행의 모든 문제를 해결하지는 않는다.

에이전트가 동시에 많은 PR을 생성하면 결국 사람이 검토해야 한다. 실행 가능한 에이전트의 수가 무한히 늘어나더라도, 사람의 리뷰 대역폭은 무한하지 않다.

즉, 병목은 코드 작성 속도에서 **검토하고 이해할 수 있는 속도**로 이동한다.

## 5. Skills: 프로젝트의 암묵지를 문서로 외부화하기

AI 에이전트는 새로운 세션을 시작할 때 프로젝트의 맥락을 거의 모른다.

코드베이스를 탐색하면서 어느 정도 추론할 수는 있지만, 코드만 읽고는 알기 어려운 정보가 많다.

예를 들면 이런 것들이다.

- 이 프로젝트에서는 왜 특정 상태 관리 방식을 사용하지 않는가?
- 공통 컴포넌트를 수정할 때 반드시 확인해야 하는 화면은 어디인가?
- API 타입은 직접 작성하는가, 자동 생성하는가?
- Tailwind의 arbitrary value를 허용하는가?
- 폼 상태는 서버 상태와 어떻게 분리하는가?
- 테스트가 느리더라도 반드시 유지해야 하는 핵심 시나리오는 무엇인가?

팀에서 일하다 보면 이런 맥락은 대부분 사람의 머릿속, 과거 PR 댓글, 슬랙 대화에 흩어져 있다.

Skills는 이 정보를 에이전트가 반복적으로 사용할 수 있는 형태로 정리한 문서다.

단순히 “이 명령어를 실행하세요”를 적는 것이 아니다. 프로젝트가 어떤 판단 기준을 가지고 있는지 기록하는 것이다.

예를 들어 프론트엔드 프로젝트라면 다음과 같은 Skill을 만들 수 있다.

```markdown
# Form Migration Skill

## 목적

기존 antd Form 기반 화면을 react-hook-form으로 점진적으로 이전한다.

## 원칙

- 화면 전체를 한 번에 교체하지 않는다.
- 서버 요청 타입과 폼 내부 타입을 분리한다.
- 기존 validation 메시지는 변경하지 않는다.
- 저장 버튼의 활성화 조건을 반드시 회귀 테스트한다.

## 검증

- pnpm test form
- pnpm lint
- 주요 고객 설정 케이스 3개 확인
```

이렇게 하면 에이전트는 매번 프로젝트의 의도를 새로 추측하지 않아도 된다.

Skills의 중요한 효과는 생산성보다도 **의도 부채를 줄이는 것**이다.

문서화되지 않은 의도는 매번 다시 설명해야 한다. 설명하지 않으면 에이전트는 빈칸을 자신 있게 추론한다. Skills는 반복되는 설명 비용과 추론 오류를 함께 줄인다.

이 지점은 AI를 사용하지 않는 팀에도 유효하다.

에이전트가 이해하지 못하는 프로젝트 지식은, 높은 확률로 새로 합류한 개발자도 이해하기 어렵다. 에이전트를 위한 Skill을 만들다 보면 팀의 암묵지가 드러난다.

## 6. Plugins and Connectors: 코드베이스 바깥의 실제 업무와 연결하기

파일 시스템만 볼 수 있는 에이전트는 제한적인 일만 할 수 있다.

실제 개발 업무는 저장소 바깥에 흩어져 있다.

- Linear, Jira, GitHub Issues에 요구 사항이 있다.
- Slack에 추가 맥락이 있다.
- Sentry에 런타임 오류가 있다.
- Datadog에 성능 지표가 있다.
- Figma에 디자인 변경 사항이 있다.
- 내부 API와 데이터베이스에 운영 상황이 있다.

Connector는 에이전트가 이런 도구에 접근할 수 있게 한다.

이때 루프의 역할이 달라진다.

파일만 읽을 수 있는 에이전트는 수정안을 제안하는 데 그친다.

도구와 연결된 루프는 문제를 찾고, 티켓을 읽고, 코드를 수정하고, PR을 만들고, CI 결과를 확인하고, 티켓 상태를 바꾸고, 필요한 경우 채널에 알림을 보낼 수 있다.

다만 연결할 수 있다는 것과 연결해야 한다는 것은 다르다.

루프가 접근할 수 있는 범위가 넓어질수록 실수의 영향도 커진다.

초기에는 읽기 권한만 주는 것이 좋다. 이후 신뢰할 수 있는 작업부터 제한적으로 쓰기 권한을 열어야 한다.

예를 들면 이런 순서다.

1. 오류와 티켓을 읽고 요약한다.
2. 코드 수정안을 작성한다.
3. Draft PR을 생성한다.
4. 사람이 승인한 경우에만 티켓 상태를 변경한다.
5. 충분히 안정화된 뒤 제한적인 자동 머지를 검토한다.

자동화 수준은 에이전트의 능력보다 **실패했을 때 감당할 수 있는 범위**를 기준으로 정해야 한다.

## 7. Sub-agents: 작성자와 검증자를 분리하기

루프에서 특히 중요한 구조는 코드를 작성하는 에이전트와 검증하는 에이전트를 분리하는 것이다.

사람도 자신이 작성한 코드의 문제를 놓치기 쉽다. 에이전트도 마찬가지다.

코드를 작성한 에이전트는 자신이 선택한 접근을 정당화하는 방향으로 결과를 해석할 가능성이 있다.

따라서 역할을 나누는 것이 좋다.

- Explorer: 관련 코드를 탐색하고 변경 범위를 정리한다.
- Implementer: 실제 코드를 수정한다.
- Reviewer: 요구 사항과 프로젝트 규칙을 기준으로 변경 사항을 검토한다.
- Verifier: 테스트, 린트, 빌드 결과를 확인한다.

예를 들어 Reviewer에게는 다음과 같은 질문을 맡길 수 있다.

> 기존 동작이 깨질 가능성이 있는가?
>
> 요구 사항과 무관한 변경이 섞여 있는가?
>
> 타입 안정성을 약화한 부분이 있는가?
>
> 테스트가 실제 실패 가능성을 검증하는가?
>
> 프로젝트의 Skill에 정의된 규칙을 어긴 부분이 있는가?

여기서 중요한 것은 “에이전트를 많이 쓰자”가 아니다.

두 번째 의견이 필요한 지점에만 비용을 쓰는 것이다.

버튼 텍스트 수정처럼 영향 범위가 작은 작업과, 인증 흐름이나 결제 로직 변경처럼 위험도가 높은 작업에 동일한 수준의 검증 비용을 지출할 필요는 없다.

루프 설계는 결국 **위험도에 따라 검증 강도를 조절하는 작업**이다.

## 8. Memory: 대화가 아니라 저장소가 기억하게 만들기

오래 실행되는 루프에는 외부 기억 장치가 필요하다.

에이전트의 대화 컨텍스트는 영구적이지 않다. 실행이 종료되면 이전에 무엇을 시도했고, 무엇이 실패했고, 무엇이 남아 있는지 잊어버릴 수 있다.

그래서 상태는 대화가 아니라 저장소 외부에 남겨야 한다.

형식은 단순해도 된다.

```markdown
# Loop State

## Completed

- [x] Storybook 빌드 오류 원인 분석
- [x] Node 버전 불일치 수정
- [x] CI 통과 확인

## In Progress

- [ ] DatePicker 접근성 오류 수정

## Needs Human Review

- [ ] 디자인 시스템의 focus ring 정책 확인 필요

## Failed Attempts

- CSS selector 변경만으로는 Safari 오류가 해결되지 않음
```

Linear 보드나 GitHub Issue를 사용할 수도 있다.

중요한 것은 상태가 단일 대화 안에만 머물지 않는 것이다.

원문의 표현을 빌리면, **에이전트는 잊어도 저장소는 잊지 않는다.**

이 단순한 원칙이 반복 실행의 기반이 된다.

---

## 9. 프론트엔드 팀에서 실제로 만들 수 있는 첫 번째 루프

처음부터 모든 것을 자동화할 필요는 없다.

오히려 범위를 좁히는 것이 좋다.

예를 들어 프론트엔드 팀에서 비교적 안전하게 시작할 수 있는 루프는 다음과 같다.

### 예시: UI 회귀 위험 탐지 루프

매일 오전 자동화가 실행된다.

1. 전날 머지된 PR 목록을 읽는다.
2. 공통 컴포넌트, 디자인 토큰, 라우팅, 폼 로직 변경 여부를 확인한다.
3. 영향 범위가 큰 변경만 추린다.
4. 관련 Storybook 스토리와 테스트 존재 여부를 확인한다.
5. 테스트가 부족하면 별도 Worktree에서 보완안을 작성한다.
6. 다른 에이전트가 과도한 테스트 추가나 잘못된 가정을 검토한다.
7. Draft PR을 생성한다.
8. 사람이 최종 판단한다.

이 루프의 장점은 운영 코드에 바로 손대지 않는다는 것이다.

자동 머지를 하지 않아도 충분히 가치가 있다. 사람이 놓치기 쉬운 변경을 먼저 선별하고, 검토 가능한 초안을 만들어주는 것만으로도 반복 업무가 줄어든다.

다른 예시도 있다.

- 접근성 속성이 누락된 UI를 찾는 루프
- 번들 크기가 증가한 원인을 분석하는 루프
- 오래된 feature flag를 찾는 루프
- 디자인 시스템을 우회한 스타일 코드를 탐지하는 루프
- Sentry 오류 중 재현 가능한 문제를 분류하는 루프
- 의존성 업데이트 후 주요 화면의 회귀 가능성을 확인하는 루프

핵심은 “무엇이든 자동화하자”가 아니다.

다음 세 가지 조건을 만족하는 작업을 찾는 것이다.

- 반복적으로 발생한다.
- 완료 조건을 어느 정도 명확하게 정의할 수 있다.
- 실패했을 때 사람이 검토할 수 있다.

---

## 10. 루프가 해결하지 못하는 문제

루프 엔지니어링은 개발자를 제거하지 않는다.

오히려 기존에 감춰져 있던 문제를 더 선명하게 만든다.

### 첫째, 검증은 여전히 개발자의 책임이다

테스트가 통과했다는 사실과 제품이 올바르게 동작한다는 사실은 다르다.

에이전트가 “완료했다”고 말하는 것은 증명이 아니라 주장이다.

특히 프론트엔드에서는 정적 분석과 테스트만으로 잡기 어려운 문제가 많다.

- 실제 사용 흐름에서 어색한 인터랙션
- 모바일 환경에서만 발생하는 레이아웃 문제
- 접근성 저하
- 기존 고객 설정과의 예외 케이스
- 성능 저하
- 사용자가 체감하는 복잡도 증가

루프가 코드를 생성했더라도, 배포 가능한 코드인지 판단하는 책임은 남아 있다.

### 둘째, 이해 부채가 쌓일 수 있다

루프가 빠르게 코드를 작성할수록 개발자가 직접 작성하지 않은 코드가 늘어난다.

코드는 존재하지만 팀이 왜 그렇게 동작하는지 이해하지 못하는 상태가 될 수 있다.

기술 부채가 “수정하기 어려운 코드”라면, 이해 부채는 “설명할 수 없는 코드”다.

이해하지 못한 채 머지된 변경이 누적되면 언젠가 작은 수정도 어려워진다.

따라서 루프가 만든 결과를 검토할 때는 단순히 diff를 승인하는 것으로 끝내면 안 된다.

최소한 다음 질문에는 답할 수 있어야 한다.

- 왜 이 접근을 선택했는가?
- 어떤 대안을 버렸는가?
- 실패할 가능성이 높은 지점은 어디인가?
- 문제가 발생하면 어디부터 확인해야 하는가?

### 셋째, 판단을 포기하기 쉬워진다

자동화가 편해질수록 “일단 에이전트에게 맡기자”는 태도가 생길 수 있다.

문제는 에이전트를 쓰는 것 자체가 아니다.

개발자가 스스로 의견을 형성하기 전에 결과를 받아들이는 습관이다.

루프 설계는 생각을 줄이기 위한 도구가 아니다.

반복적인 실행 비용을 줄이고, 더 중요한 판단에 집중하기 위한 도구다.

좋은 루프는 개발자의 판단을 증폭한다.

나쁜 루프는 개발자가 판단하지 않아도 되는 것처럼 착각하게 만든다.

---

## 11. 프론트엔드 개발자의 역할은 어디로 이동하는가

AI 코딩 에이전트가 발전하면 코드 작성의 일부는 점점 저렴해질 가능성이 높다.

그렇다면 개발자의 가치는 어디에 남을까?

단순히 코드를 빠르게 입력하는 능력만으로 설명하기는 어려워진다.

대신 다음 능력이 더 중요해진다.

### 문제를 적절한 크기로 자르는 능력

어떤 작업은 자동화할 수 있고, 어떤 작업은 사람이 직접 판단해야 하는지 구분해야 한다.

### 완료 조건을 정의하는 능력

“적당히 리팩터링해줘”가 아니라, 어떤 테스트가 통과해야 하고 어떤 동작은 보존해야 하는지 명확히 표현해야 한다.

### 프로젝트의 의도를 문서화하는 능력

팀의 암묵지를 Skill, 테스트, 문서로 외부화해야 한다.

### 실패 범위를 제한하는 능력

자동화를 어디까지 허용할지 정하고, 위험한 작업에는 승인 단계를 남겨야 한다.

### 결과를 이해하고 설명하는 능력

에이전트가 작성한 코드를 검토하고, 동료에게 설명하고, 운영 이슈가 발생했을 때 대응할 수 있어야 한다.

결국 개발자의 역할은 “코드를 직접 작성하는 사람”에서 “코드가 안전하게 만들어지는 시스템을 설계하는 사람”으로 조금씩 확장된다.

## 12. 당장 적용해볼 수 있는 작은 실험

루프 엔지니어링이라는 표현이 거창하게 느껴질 수 있다.

하지만 시작은 작아도 된다.

다음과 같은 작업 하나를 골라볼 수 있다.

> 매주 한 번, 저장소에서 사용하지 않는 feature flag 후보를 찾아서 목록만 작성한다.

또는 다음과 같이 시작할 수도 있다.

> 최근 일주일간 발생한 CI 실패를 분류하고, 반복적으로 발생한 원인과 대응 방법을 Markdown 파일에 기록한다.

처음에는 코드를 수정하지 않아도 된다.

문제를 찾고, 분류하고, 기록하는 읽기 전용 루프부터 만들어도 충분하다.

이후 신뢰가 쌓이면 단계적으로 확장한다.

1. 문제를 탐지한다.
2. 해결 방법을 제안한다.
3. 별도 Worktree에서 수정안을 작성한다.
4. 테스트를 실행한다.
5. Draft PR을 생성한다.
6. 사람이 승인한다.

자동화의 수준을 한 단계씩 높이면서, 각 단계에서 어떤 실패가 발생하는지 관찰하는 것이 중요하다.

---

## 마치며: Build the loop. Stay the engineer.

루프 엔지니어링은 프롬프트 엔지니어링보다 쉬운 일이 아니다.

프롬프트를 직접 입력하는 횟수는 줄어들 수 있다.

하지만 무엇을 반복할지, 어떤 기준으로 완료를 판단할지, 어떤 권한을 부여할지, 어느 지점에서 사람에게 넘길지를 더 신중하게 결정해야 한다.

같은 루프를 사용하더라도 결과는 다를 수 있다.

한 개발자는 자신이 깊이 이해하는 업무를 더 빠르게 처리하기 위해 루프를 사용한다.

다른 개발자는 업무를 이해하지 않기 위해 루프를 사용한다.

겉으로 보면 둘 다 자동화를 하고 있다.

하지만 시간이 지날수록 결과는 완전히 달라진다.

루프는 그 차이를 판단하지 못한다.

그 차이를 판단하는 것은 개발자다.

**Build the loop. Stay the engineer.**

루프를 만들되, 엔지니어로 남아 있어야 한다.
