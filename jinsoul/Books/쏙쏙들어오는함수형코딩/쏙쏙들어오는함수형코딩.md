## 왜 이 책을 읽었나

이 책을 읽고 싶었던 계기는 원티드 강의였다.

그때 한창 클린 코드에 꽂혀 있었는데, 강의에서 코드를 **액션, 계산, 데이터**로 나누는 관점을 소개했다.

한데 뭉쳐 있던 코드를 분리해보는 과정이 꽤 인상적이었다.

특히 좋았던 건 단순히 "함수를 잘게 쪼개자"가 아니라, **무엇을 조심해서 건드려야 하는지 판단하는 기준**을 준다는 점이었다.

추상화의 벽 이야기도 잠깐 나왔었다. 정확한 표현은 다 기억나지 않지만, 중간 층을 둬서 요구사항이 바뀌어도 위쪽 코드가 크게 흔들리지 않게 만드는 방식이었다.

당시에는 어렴풋이 이해하고 넘어갔는데, 이번에 책을 읽으면서 그때 들었던 개념들이 조금 더 선명해졌다.

---

## 제목의 뜻

이 책의 영어 원제는 **Grokking Simplicity**다.

- **Grokking**: 완전히 이해하다, 직관적으로 깊이 체득하다는 뜻의 슬랭
- **Simplicity**: 단순함, 간결함

그러니까 직역하면 대략 **"단순함을 몸으로 체득하다"** 정도가 된다.

한국어판 제목인 "쏙쏙 들어오는"도 꽤 잘 옮긴 편이라고 느꼈다.

여담으로 일론 머스크의 AI 이름인 **Grok**도 같은 어원이다.

표면적인 답변이 아니라, 깊이 이해한다는 의미를 담은 이름이라고 한다.

이 책도 비슷하다.

겉으로 보기 좋은 코드를 만드는 법보다, **왜 이런 구조가 유지보수에 유리한지 몸으로 이해하게 만드는 책**에 가깝다.

---

## 이 책에서 내가 뽑은 핵심

이 책을 읽고 나서 내 머릿속에 가장 강하게 남은 건 두 가지였다.

- **액션 / 계산 / 데이터 구분**
- **추상화 벽**

이 둘은 결국 같은 문제를 다른 각도에서 다룬다.

### 액션 / 계산 / 데이터 구분

이 코드를 건드려도 되는지, 건드리면 위험한지를 판단하게 해준다.

### 추상화 벽

건드려야 한다면, 그 변경이 얼마나 많은 곳까지 흔들릴지를 줄여준다.

결국 둘 다 **"코드 수정이 무서운 이유"를 해결하는 도구**다.

액션/계산/데이터는 수정 전 위험도를 판단하게 해주고, 추상화 벽은 수정했을 때 충격을 흡수해준다.

---

## 1. 액션 / 계산 / 데이터

함수형 프로그래밍에서는 코드를 크게 세 가지로 나눈다.

| 분류 | 의미 | 회의실 예약 프로젝트에서의 예 |
| --- | --- | --- |
| **액션** | 실행 시점이나 횟수에 따라 결과가 달라지거나 외부에 영향을 주는 것 | 예약 생성 요청, 예약 취소 요청, 토스트 표시, 상태 업데이트 |
| **계산** | 입력이 같으면 항상 같은 결과가 나오는 것 | 시간 유효성 검사, 예약 가능 회의실 필터링, 시간 문자열을 분으로 변환 |
| **데이터** | 실행하지 않아도 그 자체로 의미가 있는 사실 | 회의실 정보, 예약 정보, 예약 폼 값 |

이 셋을 나누는 이유는 단순하다.

복잡한 소프트웨어일수록 모든 코드를 한 덩어리로 다루면 무엇이 위험한지 판단하기 어려워진다.

액션은 외부 세계와 연결되어 있기 때문에 조심해야 한다.

반면 계산은 입력과 출력만 보면 되기 때문에 테스트하기 쉽고, 언제든 여러 번 실행해도 안전하다.

데이터는 그 둘 사이에서 상태와 사실을 담는 재료가 된다.

결국 코드를 안정적으로 만들려면 **액션을 줄이고, 계산과 데이터 쪽으로 밀어내는 것**이 중요하다.

---

## 2. 액션에서 계산을 빼내야 한다

이 책에서 가장 실용적으로 와닿았던 부분도 이거였다.

액션 안에 계산이 숨어 있으면, 계산 하나를 검증하려고 해도 액션 전체 환경을 끌고 와야 한다.

회의실 예약 화면을 예로 들면, 예약 버튼을 누르는 행위 자체는 액션이다.

하지만 그 전에 "이 예약이 유효한가?"를 판단하는 건 계산이다.

```tsx
// ❌ 액션 안에 계산이 섞인 경우
function handleReserve() {
  if (!selectedRoom) {
    toast({ description: "회의실을 선택해주세요" });
    return;
  }

  const start = timeToMinutes(form.startTime);
  const end = timeToMinutes(form.endTime);

  if (end <= start) {
    toast({ description: "종료 시간은 시작 시간보다 늦어야 합니다" });
    return;
  }

  if (end - start < 30) {
    toast({ description: "최소 30분 이상 예약해야 합니다" });
    return;
  }

  reserveRoom({
    roomId: selectedRoom.id,
    date: form.date,
    start: form.startTime,
    end: form.endTime,
    attendees: form.attendees,
  });
}
```

이 코드는 동작은 한다.

하지만 검증 로직을 테스트하려면 토스트, 폼 상태, API 호출까지 함께 봐야 한다.

계산을 분리하면 구조가 훨씬 선명해진다.

```tsx
// ✅ 계산 분리
function validateReservation(formValues, selectedRoom) {
  if (!selectedRoom) {
    return { valid: false, message: "예약할 회의실을 선택해주세요" };
  }

  if (!formValues.startTime || !formValues.endTime) {
    return { valid: false, message: "시작 시간과 종료 시간을 모두 선택해주세요" };
  }

  const start = timeToMinutes(formValues.startTime);
  const end = timeToMinutes(formValues.endTime);

  if (end <= start) {
    return { valid: false, message: "종료 시간은 시작 시간보다 늦어야 합니다" };
  }

  if (end - start < 30) {
    return { valid: false, message: "최소 30분 이상 예약해야 합니다" };
  }

  return { valid: true };
}

function handleReserve() {
  const result = validateReservation(formValues, selectedRoom);

  if (!result.valid) {
    toast({ description: result.message });
    return;
  }

  reserveRoom({
    roomId: selectedRoom.id,
    date: formValues.date,
    start: formValues.startTime,
    end: formValues.endTime,
    attendees: formValues.attendees,
  });
}
```

이렇게 나누면 좋은 점이 분명하다.

- 검증 로직은 React 없이도 테스트할 수 있다
- 예약 처리 함수는 "실행"만 담당하게 된다
- 나중에 검증 규칙이 바뀌어도 수정 위치가 명확하다

코드가 길어졌다는 느낌이 들 수는 있다.

하지만 그 대가로 얻는 건 훨씬 크다.

**테스트하기 쉽고, 재사용하기 쉽고, 이해하기 쉬워진다.**

## 3. 암묵적 입출력을 지양해야 한다

책을 읽으면서 계속 강조된 건, 계산처럼 보이지만 사실은 액션에 묶여 있는 코드를 경계해야 한다는 점이었다.

예를 들어 이런 코드는 겉보기에는 검증 함수 같지만 사실 그렇지 않다.

```tsx
// ❌ 계산 함수처럼 보이지만 암묵적 입출력이 있음
function validateReservation() {
  if (!selectedRoom) {
    toast({ description: "회의실을 선택해주세요" });
    return false;
  }

  if (!form.startTime || !form.endTime) {
    toast({ description: "시간을 선택해주세요" });
    return false;
  }

  return true;
}
```

이 함수는 입력도 명시적이지 않고, 출력도 토스트라는 부수효과에 기대고 있다.

그래서 테스트도 어렵고, 재사용도 어렵다.

반대로 입력과 출력을 분명히 하면 계산이 된다.

```tsx
// ✅ 계산과 액션 분리
function validateReservation(formValues, selectedRoom) {
  if (!selectedRoom) {
    return "예약할 회의실을 선택해주세요";
  }

  if (!formValues.startTime || !formValues.endTime) {
    return "시작 시간과 종료 시간을 모두 선택해주세요";
  }

  return null;
}

function handleReserve() {
  const error = validateReservation(formValues, selectedRoom);

  if (error) {
    toast({ description: error });
    return;
  }

  reserveRoom(...);
}
```

- 계산은 입력을 파라미터로 받는다
- 계산은 결과를 반환한다
- 액션은 그 결과를 받아서 실행한다

이렇게만 해도 코드의 성질이 훨씬 또렷해진다.

## 4. 계층형 설계

책에서는 소프트웨어를 계층으로 나누는 방식을 설명한다.

**변경되는 이유가 다른 것들을 분리하는 것.**

회의실 예약 기능에 대입해보면 대략 이렇게 볼 수 있다.

```
[ 비즈니스 규칙 ]
영업 시간 안에서만 예약 가능
최소 30분 이상 예약 가능
같은 시간대 중복 예약 불가

[ 도메인 ]
회의실, 예약, 참석 인원, 장비, 층, 시간대

[ 유틸리티 ]
시간 문자열을 분으로 변환
시간 충돌 여부 계산
리스트 필터링
```

왜 나누냐면, 바뀌는 이유가 다르기 때문이다.

- 영업 시간 정책이 바뀌면 비즈니스 규칙이 바뀐다
- 회의실에 새로운 속성이 생기면 도메인이 바뀐다
- 시간 계산 방식 버그가 있으면 유틸리티가 바뀐다

이걸 한 함수에 몰아넣으면, 요청 하나가 들어올 때마다 같은 함수를 계속 열어야 한다.

```tsx
// ❌ 다 섞인 경우
function canReserve(room, reservations, formValues) {
  const start = timeToMinutes(formValues.startTime);
  const end = timeToMinutes(formValues.endTime);

  if (start < 540 || end > 1200) return false;
  if (end - start < 30) return false;
  if (room.capacity < formValues.attendees) return false;
  if (!formValues.equipments.every((equipment) => room.equipments.includes(equipment))) return false;

  return !reservations.some((reservation) =>
    reservation.roomId === room.id &&
    start < timeToMinutes(reservation.end) &&
    end > timeToMinutes(reservation.start)
  );
}
```

반대로 나누면 변경 위치가 선명해진다.

```tsx
// 유틸리티
function timeToMinutes(time) {
  const [hour, minute] = time.split(":").map(Number);
  return hour * 60 + minute;
}

function hasTimeConflict(startA, endA, startB, endB) {
  return timeToMinutes(startA) < timeToMinutes(endB) &&
         timeToMinutes(endA) > timeToMinutes(startB);
}

// 비즈니스 규칙
function isWithinBusinessHours(startTime, endTime) {
  return timeToMinutes(startTime) >= 9 * 60 &&
         timeToMinutes(endTime) <= 20 * 60;
}

function hasMinimumDuration(startTime, endTime) {
  return timeToMinutes(endTime) - timeToMinutes(startTime) >= 30;
}

// 도메인 규칙
function canFitAttendees(room, attendees) {
  return room.capacity >= attendees;
}

function hasRequiredEquipments(room, equipments) {
  return equipments.every((equipment) => room.equipments.includes(equipment));
}
```

이렇게 나누면 좋은 점은 함수를 여러 개로 쪼갰다는 데 있지 않다. 진짜 차이는 변경 이유가 코드 구조에 드러난다는 데 있다.

예를 들어 "최소 예약 시간을 바꿔주세요"는 정책의 문제이고, "시간 계산에 버그가 있어요"는 구현의 문제다.

이 둘이 한 함수 안에 섞여 있으면 같은 코드를 함께 건드려야 하지만, 계층을 나누면 어디를 수정해야 하는지가 바로 보인다.

결국 계층형 설계의 장점은 코드를 잘게 나누는 것이 아니라, **변경의 종류에 따라 수정 위치와 영향 범위를 분리하는 것**이다.

## 4-1. 추상화 벽

계층형 설계를 나누는 패턴중 하나는 추상화 벽이다. 

**바뀔 가능성이 큰 세부 구현을 아래로 숨기고, 위쪽 코드는 그 사실을 몰라도 되게 만드는 것**이다.

즉, 중요한 건 함수를 나눴느냐가 아니라,

**아래 구현이 바뀌어도 위 코드는 안 바뀌는가**다.

회의실 예약 기능으로 생각해보면 이 개념이 조금 더 분명해진다.

예를 들어 화면이 회의실 목록을 보여줘야 한다고 하자.

추상화 벽이 없으면 화면이 API 주소도 알고, 응답 필드 이름도 알고, 그 응답을 화면에서 쓸 수 있는 형태로 바꾸는 일도 직접 하게 된다.

```tsx
function AvailableRoomsSection() {
  const [rooms, setRooms] = useState([]);

  useEffect(() => {
    fetch("/api/rooms")
      .then((res) => res.json())
      .then((data) => {
        setRooms(
          data.map((room) => ({
            id: room.room_id,
            name: room.room_name,
            floor: room.room_floor,
            capacity: room.max_capacity,
            equipments: room.facilities,
          })),
        );
      });
  }, []);

  return (
    <ul>
      {rooms.map((room) => (
        <li key={room.id}>{room.name}</li>
      ))}
    </ul>
  );
}
```

이 구조에서는 화면 코드가 너무 많은 걸 알고 있다.

- API 주소가 무엇인지
- 응답 필드 이름이 무엇인지
- 서버 데이터와 화면 데이터의 차이가 무엇인지

이 상태에서는 서버 응답이 조금만 바뀌어도 화면 코드가 같이 흔들린다.

추상화 벽을 세우면 이 책임을 아래로 내릴 수 있다.

```tsx
function normalizeRoom(rawRoom) {
  return {
    id: rawRoom.room_id,
    name: rawRoom.room_name,
    floor: rawRoom.room_floor,
    capacity: rawRoom.max_capacity,
    equipments: rawRoom.facilities,
  };
}

async function fetchRooms() {
  const response = await fetch("/api/rooms");
  const data = await response.json();
  return data.map(normalizeRoom);
}
```

그러면 화면은 이렇게 바뀐다.

```tsx
function AvailableRoomsSection() {
  const [rooms, setRooms] = useState([]);

  useEffect(() => {
    fetchRooms().then(setRooms);
  }, []);

  return (
    <ul>
      {rooms.map((room) => (
        <li key={room.id}>{room.name}</li>
      ))}
    </ul>
  );
}
```

이제 화면은 그냥 "회의실 목록을 가져온다"만 안다.

실제 API 주소가 무엇인지, 응답이 어떤 모양인지, 어떤 필드를 어떤 이름으로 바꿔야 하는지는 더 이상 신경 쓰지 않는다.

이게 추상화 벽이다.

이 구조의 진짜 장점은 변경이 생길 때 드러난다.

예를 들어 서버 응답이 다음처럼 바뀌었다고 해보자.

```tsx
function normalizeRoom(rawRoom) {
  return {
    id: rawRoom.id,
    name: rawRoom.name,
    floor: rawRoom.level,
    capacity: rawRoom.seats,
    equipments: rawRoom.equipments,
  };
}
```

이때 바뀌는 건 `normalizeRoom` 같은 벽 아래 코드뿐이다.

화면 코드는 그대로 둘 수 있다.

즉, **변경의 충격을 아래에서 흡수한 것**이다.

내가 이해한 추상화 벽은 이런 것이다.

코드를 예쁘게 감추는 기술이 아니라, **자주 바뀔 가능성이 있는 세부사항을 한곳에 가두는 기술**.

그래서 요구사항이 바뀌었을 때, 얼마나 많은 곳이 흔들릴지를 줄여주는 장치다.

## 마무리

솔직히 말하면, 책을 읽는 동안 엄청난 깨달음이 연속으로 오는 책은 아니었다.

그런데 다 읽고 나서 내 코드를 다시 보니까 오히려 책이 말한 핵심이 더 잘 보였다.

회의실 예약 기능에서

- 예약 생성과 취소는 액션이고
- 시간 검증과 예약 가능 여부 판정은 계산이고
- 회의실 정보와 예약 정보는 데이터다

이렇게 나눠서 보기 시작하니까,

어떤 코드를 테스트하기 어려운 이유가 보였고,

어떤 로직이 UI에 너무 가까이 붙어 있는지도 보였고,

어떤 부분에 추상화 벽이 더 필요할지도 보였다.

결국 이 책이 말하는 단순함은 코드가 짧아지는 상태가 아니다.

**변경이 들어왔을 때 어디를 건드려야 하는지가 분명해지는 상태**에 가깝다.

아마 Grokking이라는 단어도 그래서 붙은 것 같다.

읽는 순간 이해하는 게 아니라, 내 코드에 적용해보다가

"아, 이게 그 말이었구나" 하고 뒤늦게 체득하게 되는 책.

그래서 이 책은 다 읽고 끝나는 책이라기보다,

**다 읽고 나서부터 시작되는 책**에 더 가깝다고 느꼈다.
